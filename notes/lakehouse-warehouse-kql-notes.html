The Lakehouse

A unified platform that combines the flexibility, scale and low-cost storage of a data lake with the structure and fast SQL query performance of a data warehouse — built on open Delta Lake format on top of OneLake.

OneLake Delta Lake Spark Open Format
01

What Is a Lakehouse?

A Lakehouse is a single, unified data architecture item in Microsoft Fabric that stores all your data — structured and unstructured — in one place, backed by OneLake (Fabric's tenant-wide data lake, "OneDrive for data").

It merges two worlds that used to be separate:

Raw/unstructured data — CSV, JSON, Parquet, images, PDFs, sub-folders. Explored like a file system.
🗄️

Managed, structured Delta tables automatically registered in the SQL analytics endpoint & Spark catalog.
🔗

Virtual pointers to data in ADLS Gen2, S3, or other Fabric items — no data duplication or movement.

Auto-generated, read-only T-SQL query surface over every Delta table in the Lakehouse.
💡
Key idea: One copy of data, many engines. Spark, T-SQL, Power BI (via Direct Lake mode), and KQL can all query the same Delta tables without copying data around.
02

Architecture

Every Fabric Lakehouse item is physically stored inside a workspace on OneLake, using the open-source Delta Parquet format under the hood.

OneLake — the foundation

OneLake is a single, logical data lake for the whole organization. Every Fabric workspace gets a folder in OneLake automatically — no separate storage account to provision, no keys to manage.

LayerPurpose
Bronze / FilesLanding zone for raw ingested data, unprocessed, any format.
Silver / Tables (cleansed)Cleaned, validated, deduplicated Delta tables.
Gold / Tables (curated)Business-ready, aggregated, modeled Delta tables consumed by BI & ML.
📘
This Bronze → Silver → Gold pattern is the medallion architecture, the standard way to organize a Lakehouse.

03

Creating & Loading Data

A Lakehouse is created inside a workspace: + New item → Lakehouse. It automatically provisions the Files section, Tables section, a SQL analytics endpoint, and a default semantic model.

pyspark — write a Delta table
# Read raw files landed in the Files section
df = spark.read.format("csv").option("header", True).load("Files/raw/sales/2026/")

# Basic cleansing
df_clean = df.dropDuplicates().na.drop()

# Write as a managed Delta table — appears under Tables/
df_clean.write.format("delta") \
    .mode("overwrite") \
    .saveAsTable("sales_silver")

Ways to load data

  • Upload — drag & drop files directly into the Files section.
  • Notebooks — PySpark, Scala, Spark SQL, or R code to ingest & transform.
  • Dataflows Gen2 — low-code Power Query style ingestion.
  • Data Factory pipelines — orchestrated, scheduled copy activities from 100+ connectors.
  • Shortcuts — zero-copy links to existing external/internal data.
04

Delta Lake Tables & Spark

Delta Lake is an open-source storage layer that adds ACID transactions, schema enforcement, versioning ("time travel"), and upserts (MERGE) on top of plain Parquet files.

spark sql — table operations
-- Create a managed Delta table
CREATE TABLE customers (
  id INT, name STRING, signup_date DATE
) USING DELTA;

-- Upsert (MERGE) — the classic Delta pattern
MERGE INTO customers AS target
USING updates AS source
ON target.id = source.id
WHEN MATCHED THEN UPDATE SET *
WHEN NOT MATCHED THEN INSERT *;

-- Time travel — query a previous version
SELECT * FROM customers VERSION AS OF 3;
SELECT * FROM customers TIMESTAMP AS OF '2026-07-01';

-- Maintenance
OPTIMIZE customers ZORDER BY (id);
VACUUM customers RETAIN 168 HOURS;
⚠️
VACUUM permanently deletes old file versions past the retention window — you lose time-travel ability for that data.
05

Shortcuts & OneLake Integration

A shortcut is a symbolic link inside a Lakehouse that points to data stored elsewhere — another Lakehouse, a Warehouse, ADLS Gen2, Amazon S3, or Dataverse — without copying or moving the underlying files.

  • Internal shortcuts — point to another Lakehouse/Warehouse in the same or a different Fabric workspace.
  • External shortcuts — point to ADLS Gen2, S3, or Dataverse outside Fabric.
  • OneLake catalog — data is discoverable tenant-wide once shared, following governed access via workspace roles.
💡
Shortcuts are the reason Fabric is called a "one-copy" architecture — the same physical Parquet files can be queried from a Lakehouse, a Warehouse, and a KQL Database at once.

The Warehouse

A fully transactional, enterprise-grade data warehouse experience with full T-SQL support (DDL, DML, multi-table transactions) — while still storing data as open Delta tables on OneLake.

T-SQL MPP Engine ACID Transactions Delta Under the Hood
01

What Is a Data Warehouse?

A Fabric Warehouse is a relational data warehouse item designed for SQL-first developers. Unlike the Lakehouse SQL endpoint (read-only), a Warehouse supports full CREATE, INSERT, UPDATE, DELETE, MERGE via T-SQL, with multi-statement transactions.

🏗️
Full T-SQL Surface
DDL, DML, views, stored procedures, functions — like SQL Server / Synapse.
🔄
ACID Transactions
Multi-table, multi-statement transactions with commit/rollback.
📐
MPP Compute
Massively parallel query engine, auto-scaled, no infrastructure to manage.
🌐
Cross-item Queries
Query across Warehouse + Lakehouse tables in one T-SQL statement.
📘
Even though you interact with it entirely via T-SQL, a Warehouse still stores its tables as Delta Parquet files on OneLake — so Spark and Power BI Direct Lake can read them too.
02

03

T-SQL & Table Creation

t-sql — DDL & DML
-- Create schema & table
CREATE SCHEMA sales;
CREATE TABLE sales.orders (
  order_id  INT NOT NULL,
  cust_id   INT,
  order_dt  DATE,
  amount    DECIMAL(10,2)
);

-- Transaction with multiple statements
BEGIN TRANSACTION;
  INSERT INTO sales.orders VALUES (1, 101, '2026-07-01', 250.00);
  UPDATE sales.orders SET amount = amount * 1.1 WHERE order_id = 1;
COMMIT;

-- Cross-database / cross-item query
SELECT o.order_id, l.product_name
FROM sales.orders AS o
JOIN MyLakehouse.dbo.products AS l
  ON o.cust_id = l.cust_id;
04

Schemas, Views & Stored Procedures

t-sql — views & procs
CREATE VIEW sales.vw_monthly_totals AS
SELECT DATEPART(month, order_dt) AS mth, SUM(amount) AS total
FROM sales.orders
GROUP BY DATEPART(month, order_dt);

CREATE PROCEDURE sales.usp_add_order
  @cust_id INT, @amount DECIMAL(10,2)
AS
BEGIN
  INSERT INTO sales.orders (cust_id, order_dt, amount)
  VALUES (@cust_id, GETDATE(), @amount);
END;

RETURNS TABLE
AS RETURN SELECT 1 AS allowed
WHERE @region = USER_NAME();

CREATE SECURITY POLICY RegionFilter
ADD FILTER PREDICATE security.fn_region_predicate(region)
ON sales.orders;
06

Performance & Statistics

  • Statistics — auto-created/updated to help the optimizer pick good query plans; can also be created manually with CREATE STATISTICS.
  • Result set caching — repeated identical queries can be served from cache for faster response.
  • V-Order — a write-time Parquet optimization that improves read performance across Spark, SQL & Power BI.
  • Query insights — built-in DMVs and the Fabric monitoring hub track long-running / resource-heavy queries.
💡
Because Warehouse tables are Delta under the hood, maintenance concepts like table optimization overlap heavily with the Lakehouse — the query surface is what differs, not the storage.

The KQL Database

A high-performance database built on the Kusto engine, purpose-built for near-real-time ingestion and analysis of streaming, log, telemetry and time-series data using the Kusto Query Language (KQL).

Kusto Engine Streaming Ingest Time-Series KQL
01

What Is KQL / Kusto?

KQL (Kusto Query Language) is a read-only query language optimized for exploring large volumes of structured, semi-structured (JSON) and time-stamped data — the same engine that powers Azure Data Explorer, Log Analytics, and Application Insights.

A KQL Database in Fabric lives inside a Eventhouse (a workspace container for one or more KQL Databases) and is part of the Real-Time Intelligence experience.

⚡
Eventhouse
Top-level item hosting one or more KQL Databases with shared capacity.
🗃️
KQL Database
Contains tables, materialized views, functions & update policies.
📡
Eventstream
No-code tool to route streaming data (Kafka, Event Hubs, IoT) into a KQL Database.
📊
Real-Time Dashboards
Auto-refreshing visuals built directly on top of KQL queries.
02

KQL Databases are the storage/query engine behind Fabric's Real-Time Intelligence experience — designed for data that arrives continuously and needs to be queried within seconds of arrival.

KQL Query Syntax Basics

A KQL query starts with a table name and flows through a pipe (|) chain of operators, read top to bottom — similar in feel to a Unix pipeline.

kql — basic query
StormEvents
| where StartTime >= datetime(2026-01-01)
| where State == "TEXAS"
| project StartTime, EventType, DamageProperty
| sort by DamageProperty desc
| take 10
PieceMeaning
Table nameWhere the query starts — e.g. StormEvents
| pipePasses the result of one operator into the next
Operatorwhere, project, summarize, etc.
Case sensitivityKQL is case-sensitive — table & column names must match exactly
04

Common Operators

kql — operator cheatsheet
// Filter rows
T | where Level == "Error" and Timestamp > ago(1h)

// Select / rename columns
T | project Id, Msg = Message, Timestamp

// Add computed column
T | extend DurationMs = EndTime - StartTime

// Aggregate
T | summarize Count = count(), AvgVal = avg(Value) by bin(Timestamp, 1h)

// Join two tables
T1 | join kind=inner (T2) on Id

// Sort & limit
T | sort by Timestamp desc | take 50

// Render a chart directly from a query
T | summarize count() by bin(Timestamp, 1h) | render timechart
OperatorDoes
whereFilters rows by a boolean condition
projectSelects, renames, or computes specific columns
extendAdds new computed columns, keeping the rest
summarizeGroups & aggregates rows (count, avg, sum, percentile...)
joinCombines rows from two tables on a key
bin()Buckets a value (often a datetime) into fixed-size intervals
renderInstructs the client to draw a chart (timechart, barchart, piechart...)
05

Data Ingestion & Update Policies

Data can land in a KQL Database via streaming (Eventstream, Kafka, Event Hubs), batch (pipelines, one-click ingestion of files), or push via SDKs/REST API.

kql — table & update policy
// Create a raw landing table
.create table RawEvents (Payload: dynamic, IngestedAt: datetime)

// Create the transformed target table
.create table ParsedEvents (Id: string, Level: string, Timestamp: datetime)

// Function that reshapes raw data
.create function TransformEvents() {
  RawEvents
  | extend Id = tostring(Payload.id),
           Level = tostring(Payload.level),
           Timestamp = todatetime(Payload.ts)
  | project Id, Level, Timestamp
}

// Auto-run the function whenever RawEvents gets new data
.alter table ParsedEvents policy update
```[{
  "IsEnabled": true,
  "Source": "RawEvents",
  "Query": "TransformEvents()",
  "IsTransactional": true
}]```
💡
Update policies are Kusto's equivalent of a streaming ETL trigger — every time rows land in the source table, the function runs automatically and writes the result to the target table.
06

Use Cases & Best Practices

  • Log & telemetry analytics — application logs, infrastructure metrics, diagnostics.
  • IoT & sensor data — high-volume time-stamped device readings.
  • Security / SIEM style analysis — anomaly and pattern detection over event streams.
  • Clickstream & user behavior analytics — near-real-time engagement dashboards.
⚠️
KQL Databases are optimized for read-heavy, append-mostly, time-ordered data — they are not a substitute for an OLTP system needing frequent row-level updates/deletes.
