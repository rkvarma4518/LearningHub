dbt Core Concepts

dbt (data build tool) lets analytics engineers transform data already loaded into a warehouse by writing modular, version-controlled SELECT statements — dbt handles dependency ordering, materialization, testing, and documentation. It's the "T" in ELT.

SQL-first Version Controlled ELT Jinja Templating
01

What Is dbt?

dbt does not extract or load data — it assumes raw data already sits in your warehouse (Snowflake, BigQuery, Databricks, Redshift, Postgres, Fabric, etc). dbt's job is the transform step: it compiles Jinja+SQL models into plain SQL, runs them in dependency order, and materializes the results as tables or views.

📝
Just SELECT statements
A model is a .sql file with a single SELECT — dbt wraps it in the right DDL/DML.
🔀
Dependency graph (DAG)
ref() calls between models build an automatic execution order.
🧪
Built-in testing
Assert not_null, unique, relationships and custom business rules.
📚
Auto-generated docs
A searchable docs site with column descriptions and lineage graph.
📘
dbt Core is the free, open-source CLI. dbt Cloud adds a hosted IDE, job scheduler, and semantic layer on top of the same core engine.
02

Where dbt Fits: ELT vs ETL

Traditional ETL transforms data before loading. Modern ELT loads raw data first (via Fivetran, Airbyte, ADF, etc.) and transforms it inside the warehouse afterward — that in-warehouse transform layer is where dbt lives.

Extract
Source Systems
→
Load
Raw / Staging
→
Transform

→
Serve
BI / Analytics

Typical model layering

  • staging (stg_) — 1:1 with a source table; light renaming, casting, no joins.
  • intermediate (int_) — joins/aggregations that aren't yet final, reused by multiple marts.
  • marts (fct_ / dim_) — business-facing fact and dimension tables consumed by BI tools.
03

Project Structure

Running dbt init my_project scaffolds a standard folder layout. Everything is plain text, so the whole project lives in Git.

project layout
my_project/
├── dbt_project.yml       # project-level config
├── packages.yml          # declared package dependencies
├── profiles.yml          # connection creds (usually ~/.dbt/)
├── models/
│   ├── staging/
│   │   ├── stg_orders.sql
│   │   └── stg_customers.sql
│   ├── intermediate/
│   │   └── int_orders_joined.sql
│   ├── marts/
│   │   ├── fct_orders.sql
│   │   └── dim_customers.sql
│   └── sources.yml       # source table declarations
├── seeds/                # static CSVs loaded as tables
├── snapshots/            # SCD Type 2 history tracking
├── macros/                # reusable Jinja functions
├── tests/                # singular (custom) SQL tests
└── analyses/             # ad-hoc compiled SQL, not run/materialized
04

Models & Materializations

A model is a .sql file containing one SELECT statement. The materialization controls how dbt persists it in the warehouse.

models/marts/fct_orders.sql
{{ config(materialized='table') }}

SELECT
  o.order_id,
  o.customer_id,
  c.customer_name,
  o.order_total,
  o.order_date
FROM {{ ref('stg_orders') }} o
LEFT JOIN {{ ref('stg_customers') }} c
  ON o.customer_id = c.customer_id
MaterializationWhat it doesBest for
viewCREATE VIEW — query re-runs every time it's selectedLightweight staging models, fast iteration
tableFull CREATE TABLE AS, rebuilt completely on every runMarts that are queried often, smaller/medium datasets
incrementalOnly inserts/merges new or changed rows on subsequent runsLarge, append-heavy fact tables
ephemeralNot materialized at all — inlined as a CTE into downstream modelsSmall reusable logic that shouldn't clutter the warehouse
💡
Set a sensible default in dbt_project.yml (e.g. view for staging, table for marts) and override per-model with a config() block only where needed.
05

dbt_project.yml & Config Hierarchy

Config can be set at the project level, folder level, or individual model level — the most specific setting wins.

dbt_project.yml
name: 'my_project'
version: '1.0.0'
profile: 'my_project'

model-paths: ["models"]
seed-paths: ["seeds"]
snapshot-paths: ["snapshots"]

models:
  my_project:
    staging:
      +materialized: view
    marts:
      +materialized: table
      +schema: marts

Page 02 / Building Blocks

Modeling & Jinja

ref(), source(), Jinja control flow, macros, seeds, snapshots and packages — the reusable building blocks that turn plain SQL into a maintainable, DRY transformation layer.

ref() / source() Jinja Macros Snapshots
01

ref() & source()

Never hardcode a table name. ref() points to another dbt model; source() points to a raw table declared in a sources.yml file. Both let dbt build the dependency graph and rewrite the correct fully-qualified name for the target environment.

models/sources.yml
version: 2

sources:
  - name: raw
    database: analytics_raw
    schema: ecommerce
    tables:
      - name: orders
      - name: customers
models/staging/stg_orders.sql
SELECT
  id AS order_id,
  customer_id,
  status,
  created_at AS order_date
FROM {{ source('raw', 'orders') }}
⚠️
A model referenced with ref() that is never actually built, or a source() table that doesn't exist, will fail at dbt compile / dbt run — this is what makes the DAG reliable.
02

Jinja Basics

dbt compiles Jinja ({{ }} expressions, {% %} statements) into raw SQL before sending it to the warehouse. This gives you variables, loops, conditionals and functions inside SQL.

jinja control flow
-- set a variable
{% set payment_methods = ['credit_card', 'paypal', 'bank_transfer'] %}

SELECT
  order_id,
  {% for method in payment_methods %}
  SUM(CASE WHEN payment_method = '{{ method }}' THEN amount END) AS {{ method }}_total
  {% if not loop.last %},{% endif %}
  {% endfor %}
FROM {{ ref('stg_payments') }}
GROUP BY 1

Common built-in variables

  • {{ this }} — the current model's own database relation (used mostly in incremental logic).
  • {{ target }} — info about the active connection: target.name, target.schema.
  • {{ var('name') }} — reads a variable passed via dbt_project.yml or --vars.
  • {{ env_var('NAME') }} — reads an OS environment variable, commonly for secrets.
03

Macros

Macros are reusable Jinja functions defined once in macros/ and called from any model — the SQL equivalent of a function, keeping logic DRY.

macros/cents_to_dollars.sql
{% macro cents_to_dollars(column_name) %}
    ({{ column_name }} / 100)::numeric(16, 2)
{% endmacro %}

-- usage inside a model --
SELECT {{ cents_to_dollars('amount_cents') }} AS amount_usd
FROM {{ ref('stg_payments') }}
💡
The dbt_utils package ships dozens of ready-made macros (date_spine, surrogate_key, pivot, star) so you rarely need to write common ones from scratch.
04

Seeds

Seeds are small, static CSV files checked into the repo (lookup tables, country codes, manual mappings) that dbt loads into the warehouse as tables via dbt seed.

terminal
dbt seed                 # load all CSVs in seeds/ as tables
dbt seed --full-refresh   # drop & recreate seed tables

Reference a seed exactly like a model — {{ ref('country_codes') }}.

05

Snapshots (SCD Type 2)

Snapshots capture how a mutable source record changes over time, preserving history even though the source table only holds the current state — a classic Slowly Changing Dimension Type 2 pattern.

snapshots/customers_snapshot.sql
{% snapshot customers_snapshot %}
{{
  config(
    target_schema='snapshots',
    unique_key='customer_id',
    strategy='timestamp',
    updated_at='updated_at'
  )
}}
SELECT * FROM {{ source('raw', 'customers') }}
{% endsnapshot %}

Run with dbt snapshot. dbt adds dbt_valid_from / dbt_valid_to columns so every historical version of a row can be queried.

06

Packages

Packages are shareable dbt projects (macros + models) installed like libraries. Declared in packages.yml and pulled with dbt deps.

packages.yml
packages:
  - package: dbt-labs/dbt_utils
    version: [">=1.0.0", "<2.0.0"]
  - package: calogica/dbt_expectations
    version: [">=0.10.0", "<0.11.0"]
  • dbt_utils — generic tests, date spines, surrogate keys, SQL helpers.
  • dbt_expectations — Great-Expectations-style tests (distribution, regex, row count).
  • codegen — auto-generate staging model & source YAML boilerplate.
Page 03 / Quality & Governance

Testing & Docs

dbt treats data quality as code: generic and singular tests catch broken assumptions before they reach a dashboard, and auto-generated documentation keeps lineage and column meaning discoverable.

Generic Tests Singular Tests Docs Site Freshness
01

Generic (Schema) Tests

Generic tests are parameterized and reusable — applied declaratively in a model's YAML file. Four ship built into dbt Core.

models/marts/schema.yml
version: 2

models:
  - name: fct_orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: customer_id
        tests:
          - relationships:
              to: ref('dim_customers')
              field: customer_id
      - name: status
        tests:
          - accepted_values:
              values: ['placed', 'shipped', 'returned']
TestChecks
uniqueNo duplicate values in the column
not_nullNo NULLs in the column
accepted_valuesEvery value is one of an allowed list
relationshipsReferential integrity — every value exists in a related model
02

Singular & Custom Generic Tests

A singular test is a one-off SQL file in tests/ that returns failing rows — if it returns any rows, the test fails.

tests/assert_positive_order_totals.sql
-- fails if any row is returned
SELECT order_id, order_total
FROM {{ ref('fct_orders') }}
WHERE order_total < 0

A custom generic test is a parameterized macro in macros/ prefixed test_, reusable across many columns just like the built-ins.

📘
Run all tests with dbt test, or scope to one model with dbt test --select fct_orders. Set severity: warn on a test to log rather than fail the build.
03

Documentation

Descriptions live alongside tests in YAML. dbt docs generate builds a static site with the compiled SQL, column descriptions, and full lineage — served locally via dbt docs serve.

models/marts/schema.yml
models:
  - name: fct_orders
    description: "One row per order, at grain of order_id."
    columns:
      - name: order_id
        description: "Primary key — unique order identifier."

Longer, reusable descriptions can live in a .md file using {% docs %} blocks and be referenced with {{ doc('block_name') }}.

04

The DAG & Lineage Graph

Every ref() and source() call becomes an edge in dbt's dependency graph. The docs site renders this visually, and it also drives selective running.

selective run syntax
dbt run --select fct_orders          # just this model
dbt run --select +fct_orders         # this model + everything upstream
dbt run --select fct_orders+         # this model + everything downstream
dbt run --select staging.*           # everything in the staging folder
dbt run --select tag:nightly         # everything tagged 'nightly'
05

Source Freshness

dbt can assert that a raw source table has been loaded recently enough, based on a timestamp column — useful for catching a broken upstream EL pipeline early.

sources.yml — freshness block
sources:
  - name: raw
    tables:
      - name: orders
        loaded_at_field: _loaded_at
        freshness:
          warn_after: {count: 12, period: hour}
          error_after: {count: 24, period: hour}

Checked with dbt source freshness.

06

Exposures

Exposures declare downstream consumers — a dashboard, ML model, or app — so the lineage graph shows the full path from raw source to business usage, and impact analysis catches breaking changes before they hit a report.

models/exposures.yml
exposures:
  - name: executive_revenue_dashboard
    type: dashboard
    depends_on:
      - ref('fct_orders')
    owner:
      name: Analytics Team
💡
dbt Cloud's Semantic Layer builds on this same metadata (metrics defined once, consumed consistently across every BI tool).

01

Incremental Models Deep Dive

An incremental model only processes new/changed rows after the first full build, using {{ is_incremental() }} to conditionally filter the SELECT.

models/marts/fct_events.sql
{{
  config(
    materialized='incremental',
    unique_key='event_id',
    incremental_strategy='merge'
  )
}}

SELECT *
FROM {{ source('raw', 'events') }}

{% if is_incremental() %}
  WHERE event_timestamp > (SELECT MAX(event_timestamp) FROM {{ this }})
{% endif %}
StrategyBehavior
appendSimply inserts new rows — no updates, can create duplicates
mergeUpserts on unique_key — most warehouses support this
delete+insertDeletes matching rows in the target then inserts new — common on Postgres/Redshift
⚠️
Run dbt run --full-refresh whenever you change the model's column set, logic in a materially breaking way, or need to rebuild history from scratch.
02

Hooks & Operations

Hooks run SQL at specific points in the run cycle — useful for grants, logging, or vacuuming.

hook types
-- in dbt_project.yml or a model config --
pre-hook:  "DELETE FROM {{ this }} WHERE loaded_date = current_date"
post-hook: "GRANT SELECT ON {{ this }} TO ROLE reporting"
on-run-start: "..."   # fires once at the start of dbt run
on-run-end:   "..."   # fires once at the end of dbt run

A one-off DDL/DML statement outside the normal DAG can be run directly with dbt run-operation macro_name.

03

Environments, profiles.yml & Targets

profiles.yml (kept outside the repo, usually ~/.dbt/) holds warehouse connection details per target — typically dev vs prod.

~/.dbt/profiles.yml
my_project:
  target: dev
  outputs:
    dev:
      type: snowflake
      account: xy12345
      user: {{ env_var('DBT_USER') }}
      schema: dbt_dev_jsmith
    prod:
      type: snowflake
      account: xy12345
      user: {{ env_var('DBT_USER') }}
      schema: analytics
📘
Each developer typically gets their own dev schema so local iteration never collides with teammates or production.
04

Deployment & CI/CD

  • Slim CI — run/test only models changed vs. a base branch using --select state:modified+ and a stored manifest.json.
  • Scheduled jobs — dbt Cloud jobs, Airflow, Dagster, or GitHub Actions trigger dbt build on a cadence.
  • dbt build — runs models, tests, seeds, and snapshots together in DAG order, stopping downstream models if a test fails.
  • Blue/green or WAP — write-audit-publish patterns validate a build in a staging schema before swapping it into production.
typical CI job
dbt deps
dbt seed --target ci
dbt build --select state:modified+ --target ci

CLI Command Cheatsheet

core commands
dbt init <project>         # scaffold a new project
dbt deps                     # install packages from packages.yml
dbt debug                    # test the warehouse connection
dbt compile                  # render Jinja to SQL, no execution
dbt run                       # build all models
dbt run --select fct_orders   # build one model + selectors
dbt test                      # run all tests
dbt seed                      # load seed CSVs
dbt snapshot                  # run all snapshots
dbt build                     # run + test + seed + snapshot, in DAG order
dbt docs generate              # build the docs site
dbt docs serve                 # serve docs locally
dbt source freshness           # check source freshness
dbt run-operation <macro>      # run a macro directly
dbt ls --select tag:nightly    # list resources matching a selector
