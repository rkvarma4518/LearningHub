Introduction to Git

Git is a distributed version control system (DVCS) created by Linus Torvalds in 2005 to manage the Linux kernel source code. Every developer has a full local copy of the repository, including its complete history.

⚡
Fast
Most operations are local, so they're blazingly fast — no network needed.
🛡️
Integrity
Everything is checksummed with SHA-1. Corruption is instantly detectable.
🌿
Branching
Cheap, fast branches encourage parallel development and experimentation.
🔄
Distributed
Every clone is a full backup. No single point of failure.

VCS vs DVCS

FeatureCentralized (SVN)Distributed (Git)
Repository copiesSingle central serverEvery developer has full copy
Offline work❌ Limited✅ Full history available offline
BranchingExpensive / slowCheap, instant
SpeedNetwork dependentMostly local
BackupCentral server = single pointEvery clone is a backup

First-Time Setup (git config)

Git has three levels of configuration: system (all users), global (your user), and local (this repo). Each overrides the previous.

git config 
# Set your identity (stored in ~/.gitconfig)
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# Set default branch name
git config --global init.defaultBranch main

# Set preferred editor
git config --global core.editor "code --wait"   # VS Code
git config --global core.editor "vim"            # Vim
git config --global core.editor "nano"           # Nano

# Enable colored output
git config --global color.ui auto

# Set merge tool
git config --global merge.tool vimdiff

# Set line ending handling
git config --global core.autocrlf input   # Mac/Linux
git config --global core.autocrlf true    # Windows

# Create an alias
git config --global alias.st status
git config --global alias.lg "log --oneline --graph --all"

# View all config
git config --list

# View global config file
cat ~/.gitconfig

# Set config for current repo only (local)
git config user.email "work@company.com"  # no --global flag

Core Concepts

The Three States

Every file in Git lives in one of three states. Understanding this is the foundation of Git.

Working Dir
Modified
→
Staging Area
Staged
→
.git directory
Committed
  • Working Directory — Your local filesystem. Files you can see and edit. Changes here are "untracked" or "modified".
  • Staging Area (Index) — A preparation zone. You add changes here before committing. Allows fine-grained commits.
  • Repository (.git) — The actual Git database. git commit saves the staged snapshot permanently here.

Snapshots, Not Diffs

Unlike SVN (which stores file differences), Git stores snapshots of the entire project at each commit. If a file hasn't changed, Git just stores a pointer to the previous identical file. This makes branching and merging fast.

HEAD

HEAD is a special pointer that tells Git which branch (and therefore which commit) you're currently on. It's always pointing to the latest commit of your current branch. When you checkout a branch, HEAD moves to that branch.

HEAD examples 
# HEAD → main → latest commit
cat .git/HEAD
ref: refs/heads/main

# Detached HEAD (pointing directly to a commit)
git checkout abc1234
You are in 'detached HEAD' state...

# HEAD~ means "parent of HEAD"
# HEAD~2 means "grandparent of HEAD"
# HEAD^ is same as HEAD~1

SHA-1 Hash

Every object in Git (commit, tree, blob, tag) gets a unique 40-character SHA-1 hash. You can refer to any commit using its full hash or just the first 7+ characters.

💡
Tip: You never need to type the full 40-char hash. Git accepts any unambiguous prefix — usually 7–8 characters is enough (e.g., git show a3f2c1b).

Init & Staging

Starting a repository 
# Initialize a new repo in current directory
git init
Initialized empty Git repository in /project/.git/

# Initialize with a specific folder name
git init my-project

# Check status (your most-used command)
git status
On branch main
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        index.html

# Short status output
git status -s
?? index.html
M  app.js
A  newfile.txt

git add — Staging Files

git add 
# Stage a specific file
git add index.html

# Stage multiple files
git add file1.js file2.css

# Stage all changes in current directory
git add .

# Stage all tracked modified files (not untracked)
git add -u

# Stage all files (tracked + untracked)
git add -A

# Interactively stage parts of a file (hunks)
git add -p filename.js
# Options: y=yes, n=no, s=split, e=edit, q=quit

# Unstage a file (keep changes in working dir)
git restore --staged index.html
# or older syntax:
git reset HEAD index.html

# Discard working directory changes (DANGER: irreversible)
git restore index.html
# or older syntax:
git checkout -- index.html
⚠️
Difference: git add . vs git add -A
git add . stages all changes in the current directory and below. git add -A stages all changes in the entire repository, regardless of your current directory.
05

Commits

A commit is a permanent snapshot of staged changes. Each commit records: author, email, timestamp, message, and a pointer to the parent commit(s).

git commit 
# Commit staged changes (opens editor for message)
git commit

# Commit with inline message
git commit -m "feat: add login button"

# Stage all tracked files and commit in one step
git commit -am "fix: correct typo in header"

# Amend the last commit (message or staged content)
git commit --amend -m "feat: updated message"
# WARNING: Don't amend pushed commits (rewrites history)

# Amend without changing the commit message
git commit --amend --no-edit

# Empty commit (useful for triggering CI)
git commit --allow-empty -m "ci: trigger pipeline"

Conventional Commits

A widely adopted commit message standard that makes history readable and enables automated changelogs:

Commit message format 
# Format: <type>(scope): <description>

feat(auth): add OAuth2 login support
fix(cart): prevent duplicate items on refresh
docs(readme): update installation steps
style(button): fix alignment on mobile
refactor(api): simplify response parsing
test(auth): add unit tests for login
chore(deps): upgrade react to 18.3
perf(search): add database index for faster queries
revert: revert "feat(auth): add OAuth2 login"

# Breaking change (adds ! after type)
feat(api)!: remove deprecated /v1/users endpoint

Branching

A branch is just a lightweight movable pointer to a commit. Creating a branch is instant because Git only creates a new pointer file — it doesn't copy any files.

A---B---C feature/login / main----1---2---3---4 ← HEAD (main) \ X---Y feature/payment
Branch commands 
# List all local branches
git branch

# List all branches including remote
git branch -a

# List remote branches only
git branch -r

# Create a new branch
git branch feature/login

# Switch to a branch
git switch feature/login
# or (older syntax)
git checkout feature/login

# Create and switch in one step ✨
git switch -c feature/payment
# or (older syntax)
git checkout -b feature/payment

# Create branch from specific commit
git switch -c hotfix/bug abc1234

# Rename current branch
git branch -m new-name

# Delete a merged branch
git branch -d feature/login

# Force delete (even if unmerged)
git branch -D feature/abandoned

# See which branches are merged into current
git branch --merged

# See unmerged branches
git branch --no-merged

Trunk-BasedDevOps / high velocityEveryone commits to main; feature flags for control

Merging & Rebase

git merge

Merging 
# Merge feature branch INTO main
git switch main
git merge feature/login

# Fast-forward merge (linear history, no merge commit)
git merge --ff-only feature/login

# Always create a merge commit (even if FF possible)
git merge --no-ff feature/login

# Squash all branch commits into one on target
git merge --squash feature/login
git commit -m "feat: login feature"

# Abort a merge in progress
git merge --abort

Handling Merge Conflicts

Conflict resolution 
# When conflict occurs, file looks like this:
<<<<<<< HEAD
console.log("Hello from main");
=======
console.log("Hello from feature");
>>>>>>> feature/login

# 1. Open file and resolve manually, then:
git add resolved-file.js
git commit

# Use a merge tool
git mergetool

# See conflicts
git diff --diff-filter=U

git rebase

Rebase moves or replays your commits onto another branch, creating a linear history. The commits get new SHA hashes.

Before rebase: D---E feature / main--A---B---C After: git switch feature && git rebase main main--A---B---C---D'---E' feature (linear!)
git rebase 
# Rebase current branch onto main
git switch feature/login
git rebase main

# Interactive rebase — rewrite last 3 commits
git rebase -i HEAD~3
# Options in editor: pick, squash, fixup, reword, drop, edit

# Abort rebase
git rebase --abort

# Continue after resolving conflict
git rebase --continue

# Skip a conflicting commit
git rebase --skip
🚨
Golden Rule of Rebase: Never rebase commits that have been pushed to a shared branch. Rebase rewrites history (new SHA hashes), which breaks everyone else's copies.
MergeRebase
HistoryPreserves full historyCreates linear history
Merge commitYes (with --no-ff)No
Safe on shared branch✅ Always safe❌ Dangerous
Use whenMerging into mainUpdating feature branch

Stash & Clean

git stash temporarily shelves changes so you can switch context without committing. Think of it as a clipboard for your uncommitted work.

git stash 
# Stash current changes
git stash

# Stash with a descriptive message
git stash push -m "WIP: login form validation"

# Stash including untracked files
git stash push -u

# Stash including ignored files too
git stash push -a

# List all stashes
git stash list
stash@{0}: WIP: login form validation
stash@{1}: On main: quick experiment

# Apply latest stash (keeps stash)
git stash apply

# Apply specific stash
git stash apply stash@{1}

# Apply and remove latest stash
git stash pop

# See what's in a stash
git stash show -p stash@{0}

# Create a branch from a stash
git stash branch feature/test stash@{0}

# Delete a specific stash
git stash drop stash@{1}

# Delete all stashes
git stash clear

git clean

git clean — remove untracked files 
# Preview what would be removed (dry run)
git clean -n

# Remove untracked files
git clean -f

# Remove untracked files AND directories
git clean -fd

# Remove ignored files too
git clean -fX

# Interactive mode
git clean -i

Undo & Reset

CommandWorking DirStagingCommitsSafe for shared?
git restore fileDiscards——✅
git restore --stagedKeepsUnstages—✅
git reset --soft HEAD~1KeepsKeepsUncommits❌
git reset --mixed HEAD~1KeepsUnstagesUncommits❌
git reset --hard HEAD~1DiscardsDiscardsUncommits❌
git revert HEADKeepsKeepsAdds new commit✅
Undo commands 
# Undo last commit, keep changes staged
git reset --soft HEAD~1

# Undo last commit, keep changes unstaged (DEFAULT)
git reset HEAD~1

# Undo last commit and discard ALL changes
git reset --hard HEAD~1

# Safely undo a commit (creates a reverting commit)
git revert HEAD
git revert abc1234        # revert a specific commit
git revert HEAD~3..HEAD   # revert a range

# Recover from accidental reset using reflog
git reflog
abc1234 HEAD@{0}: reset: moving to HEAD~1
def5678 HEAD@{1}: commit: feat: add login
git reset --hard def5678   # restore to before reset
💡
reflog is your safety net. Git's reflog tracks every movement of HEAD for ~90 days. Even after a hard reset, you can almost always recover commits using git reflog.

Tags

Tags mark specific commits as important — typically used for version releases (v1.0.0, v2.3.1). Unlike branches, tags don't move when new commits are made.

git tag 
# List all tags
git tag

# Lightweight tag (just a pointer)
git tag v1.0.0

# Annotated tag (recommended — includes message, author, date)
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag a specific commit
git tag -a v0.9.0 abc1234 -m "Beta release"

# Show tag details
git show v1.0.0

# Push a specific tag to remote
git push origin v1.0.0

# Push ALL tags to remote
git push origin --tags

# Delete a local tag
git tag -d v1.0.0

# Delete a remote tag
git push origin --delete v1.0.0

# Checkout a tag (enters detached HEAD)
git checkout v1.0.0

GitHub Overview

GitHub is a cloud-based hosting platform for Git repositories. It adds collaboration features like pull requests, issues, project boards, code review, and CI/CD on top of Git. Owned by Microsoft (since 2018).

🔒
Public Repos
Visible to everyone. Great for open source projects.
🔑
Private Repos
Only you and invited collaborators can access.
🌍
Organizations
Groups for teams with shared repos and access controls.
🤖
GitHub Actions
Automate CI/CD workflows directly in your repo (see below).

Authentication Methods

SSH setup (recommended) 
# Generate SSH key pair
ssh-keygen -t ed25519 -C "you@example.com"

# Start SSH agent and add key
eval "$(ssh-agent -s)"

# Copy public key to clipboard
cat ~/.ssh/id_ed25519.pub
# → Paste into GitHub: Settings > SSH keys > New SSH key

# Test SSH connection
ssh -T git@github.com
Hi username! You've successfully authenticated.

# Use token as your password when cloning over HTTPS

# Cache credentials (HTTPS)
git config --global credential.helper cache
git config --global credential.helper "store"  # saves to disk

# GitHub CLI (install gh)
gh auth login

Push, Pull & Fetch

commits
⬆ push
GitHub
Remote Repo

push / fetch / pull 
─── PUSH ────────────────────────────────────────

# Push current branch to origin
git push

# Push and set upstream tracking
git push -u origin main

# Push a specific branch
git push origin feature/login

# Push all branches
git push --all origin

# Push tags
git push origin --tags

# Force push (rewrites remote history — DANGEROUS)
git push --force

# Safer force push (fails if remote changed since last fetch)
git push --force-with-lease

# Delete a remote branch
git push origin --delete feature/old-branch

─── FETCH ───────────────────────────────────────

# Fetch remote changes (doesn't merge into working dir)
git fetch origin

# Fetch all remotes
git fetch --all

# Fetch and prune deleted remote branches
git fetch --prune

─── PULL ────────────────────────────────────────

# Pull = fetch + merge
git pull

# Pull with rebase instead of merge
git pull --rebase

# Pull from specific remote and branch
git pull origin main

# Always pull with rebase (configure globally)
git config --global pull.rebase true
ℹ️
fetch vs pull: git fetch downloads remote changes but doesn't apply them — you can inspect before merging. git pull fetches AND immediately merges. Prefer git fetch + review + git merge for more control.

Fork & Clone

git clone

Cloning 
# Clone via SSH (recommended)
git clone git@github.com:user/repo.git

# Clone via HTTPS
git clone https://github.com/user/repo.git

# Clone into a specific folder name
git clone git@github.com:user/repo.git my-folder

# Clone only the latest snapshot (no full history)

# Clone a specific branch
git clone -b develop git@github.com:user/repo.git

git clone --recurse-submodules git@github.com:user/repo.git

Forking Workflow

Forking creates your own copy of someone else's repo on GitHub. Used heavily in open source. Changes flow: Fork → Clone → Branch → Push to fork → Pull Request to original.

Fork workflow 
# 1. Fork on GitHub (click the Fork button)
# 2. Clone YOUR fork locally
git clone git@github.com:YOUR-USER/repo.git

# 3. Add original repo as "upstream"
git remote add upstream git@github.com:ORIGINAL-USER/repo.git

# 4. Verify remotes
git remote -v
origin    git@github.com:YOUR-USER/repo.git
upstream  git@github.com:ORIGINAL-USER/repo.git

# 5. Create a feature branch
git switch -c fix/typo-in-readme

# 6. Make changes, commit, push to YOUR fork
git push origin fix/typo-in-readme

# 7. Open a Pull Request on GitHub

# ── Keeping your fork up to date ──
git fetch upstream
git switch main
git merge upstream/main
git push origin main

Good PR Practices

  • Keep PRs small — aim for <400 lines of code for easy review
  • Descriptive title — use Conventional Commits format
  • Write a PR description — explain WHY, not just WHAT; link to issue (Closes #42)
  • Request specific reviewers — don't just assign randomly
  • Use draft PRs for work in progress to get early feedback
  • Respond to all review comments before asking for re-review

Create files in .github/ISSUE_TEMPLATE/ to provide structured forms for bug reports, feature requests, etc. GitHub will show these when someone clicks "New issue".

Collaboration on GitHub

Adding Collaborators

Go to Repository → Settings → Collaborators → Add people. For organizations, use teams for finer access control.

Branch Protection Rules

Protect important branches (like main) by going to Settings → Branches → Add rule:

  • Require pull request reviews — PRs must be approved before merging
  • Require status checks — CI must pass before merging
  • Require linear history — no merge commits, only rebases/squash
  • Restrict force pushes — prevents history rewriting
  • Require signed commits — ensures commit authenticity via GPG

Code Review

  • Comment — leave feedback, no approval decision
  • Approve — approve the PR for merging
  • Request changes — block merge until author addresses feedback

CODEOWNERS

Create .github/CODEOWNERS to auto-assign reviewers based on file paths:

.github/CODEOWNERS 
# Everything — default owner
*       @team-lead

# Frontend files
*.js    @frontend-team
*.css   @frontend-team

# Backend
/api/   @backend-team

# Docs
docs/   @tech-writer

GitHub Projects (Project Boards)

GitHub Projects are Kanban-style boards. You can organize issues, PRs, and notes into columns like Backlog → In Progress → In Review → Done. The new Projects v2 supports custom fields, grouping, and filtering.

Notifications & Watching

  • Watch a repo to receive all notifications
  • Participating only to get notified only when mentioned or assigned
  • Use @mention (@username or @team) in issues/PRs/comments

GitHub Pages

GitHub Pages lets you host a static website directly from a GitHub repository for free at username.github.io/repo-name.

Deploying to GitHub Pages 
# Option 1: Deploy from a branch (manual)
# Go to Repo → Settings → Pages → select branch (main or gh-pages) → /root or /docs

# Option 2: Create a gh-pages branch
git switch --orphan gh-pages   # empty branch
git add .
git commit -m "Initial GitHub Pages"
git push origin gh-pages

# Option 3: Use gh-pages npm package
npm install gh-pages --save-dev
# In package.json: "deploy": "gh-pages -d build"
npm run deploy

# Option 4: GitHub Actions (auto deploy on push)
# → mentioned in Advanced GitHub section

# Custom domain: add CNAME file to repo root
echo "www.mysite.com" > CNAME
git add CNAME && git commit -m "add custom domain" && git push
ℹ️
GitHub Pages supports Jekyll natively — put a _config.yml in your repo and GitHub will build your Jekyll site automatically. You can also use other static site generators (Hugo, VitePress) by deploying the build output via Actions.

Advanced GitHub Features

GitHub Actions (CI/CD)

GitHub Actions automates your software workflow — build, test, deploy — directly in GitHub. Workflows are YAML files in .github/workflows/. They trigger on events (push, PR, schedule). It's a powerful topic on its own — explore the official docs for full coverage.

.github/workflows/ci.yml — basic example 
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - run: npm ci
      - run: npm test

GitHub Gist

Gists are simple single-file (or multi-file) code snippets hosted on GitHub — like mini repos. They can be public or secret, and have their own Git history.

GitHub Releases

Releases bundle a tagged version with release notes and binary assets (e.g., compiled executables). Found in the Releases tab of a repo. Can be created from tags via the GitHub UI or CLI:

gh release 
gh release create v1.0.0 --title "v1.0.0 Launch" --notes "First stable release"
gh release upload v1.0.0 ./dist/app-linux.tar.gz
gh release list

GitHub Discussions

A forum-like feature for community conversations, Q&A, and announcements — separate from Issues (which are for actionable tasks). Enable in repo Settings.

GitHub Packages

Host and share software packages (npm, Docker, Maven, NuGet) directly in GitHub, alongside your source code. Packages are scoped to users or organizations.

GitHub Codespaces

Cloud-based development environments that spin up a full VS Code in the browser — pre-configured via .devcontainer/devcontainer.json. Start a Codespace from any repo's Code button.

GitHub Copilot

AI-powered code completion and chat built into VS Code and GitHub.com. Suggests code, explains functions, writes tests, and more — powered by large language models.

Dependabot

Automatically opens PRs to update your dependencies when vulnerabilities or new versions are detected. Configure in .github/dependabot.yml.

GitHub Security Features

  • Secret scanning — detects accidentally committed API keys and tokens
  • Code scanning — static analysis via CodeQL to find vulnerabilities
  • Security advisories — privately disclose and fix vulnerabilities before public disclosure
  • Required 2FA — enforce two-factor auth for organization members

Cherry-pick

Cherry-pick applies a specific commit (or range of commits) from any branch onto your current branch. Useful for applying hotfixes to multiple release branches.

git cherry-pick 
# Apply a specific commit to current branch
git cherry-pick abc1234

# Cherry-pick without committing (stage only)
git cherry-pick --no-commit abc1234

# Cherry-pick a range (from..to, exclusive..inclusive)
git cherry-pick abc1234..def5678

# Continue after conflict
git cherry-pick --continue

# Abort
git cherry-pick --abort

# Start bisect session
git bisect start

# Mark current commit as bad (has the bug)
git bisect bad

# Mark a known good commit
git bisect good v1.0.0

# Git checks out a middle commit — test it, then:
git bisect good  # or
git bisect bad

# Repeat until Git identifies the culprit commit
abc1234 is the first bad commit

# End bisect session
git bisect reset

# Automate with a test script
git bisect run npm test

Git Hooks

Hooks are scripts that run automatically at certain Git events. They live in .git/hooks/. Use Husky to share hooks via npm.

HookRuns WhenCommon Use
pre-commitBefore commit is createdRun linter, tests, format code
commit-msgAfter commit message is enteredValidate commit message format
pre-pushBefore pushing to remoteRun full test suite
post-mergeAfter successful mergeRun npm install
post-checkoutAfter checkout/switchUpdate dependencies
pre-rebaseBefore rebase startsWarn or prevent rebasing
.git/hooks/pre-commit 
#!/bin/sh
# Run eslint before every commit
npm run lint
if [ $? -ne 0 ]; then
  echo "❌ ESLint failed. Commit aborted."
  exit 1
fi
Husky setup (shareable hooks) 
npm install husky --save-dev
npx husky init

npm install @commitlint/cli @commitlint/config-conventional --save-dev

Submodules

Submodules let you embed one Git repository inside another at a specific commit. Used when you want to include a dependency as source code.

git submodule 
# Add a submodule
git submodule add git@github.com:user/lib.git libs/mylib

# Initialize submodules (after cloning)
git submodule init
git submodule update

# Or in one step after clone
git clone --recurse-submodules git@github.com:user/repo.git

# Update all submodules to their latest
git submodule update --remote

# Run command in each submodule
git submodule foreach git pull

# Remove a submodule
git submodule deinit libs/mylib
git rm libs/mylib
rm -rf .git/modules/libs/mylib

.gitignore

The .gitignore file tells Git which files and folders to never track. Rules apply to the directory it's in and all subdirectories.

.gitignore patterns 
# Ignore a specific file
.env
secrets.json

# Ignore all .log files
*.log

# Ignore a folder
node_modules/
dist/
.next/

# Ignore OS files
.DS_Store       # macOS
Thumbs.db       # Windows

# Ignore IDE files
.vscode/
.idea/

# But track this specific file (negation)
!.env.example

# Ignore only in root (leading slash)
/build

# Match any subdirectory
**/temp
gitignore tips 
# Stop tracking a file that's already committed
git rm --cached filename.log
git rm -r --cached node_modules/

# Check if a file is ignored
git check-ignore -v filename.log

# Global gitignore (applies to all repos)
git config --global core.excludesFile ~/.gitignore_global

https://gitignore.io
# or GitHub's templates at:
