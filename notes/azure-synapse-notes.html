Synapse Overview

Azure Synapse Analytics is a unified analytics service that brings together big data (Spark), enterprise data warehousing (SQL), data integration (pipelines), and real-time/BI reporting in a single workspace, built on Azure Data Lake Storage Gen2.

Synapse SQL Apache Spark ADLS Gen2 Pipelines
01

What Is Azure Synapse Analytics?

Synapse is Microsoft's enterprise analytics platform, launched as the evolution of Azure SQL Data Warehouse. It combines multiple compute engines under one workspace so data engineers, warehouse developers, and data scientists can work against the same lake without stitching together separate services.

🗄️
Synapse SQL
Dedicated SQL pools (provisioned MPP) & serverless SQL pools (pay-per-query) for T-SQL analytics.
🔥
Apache Spark Pools
Managed Spark clusters for big-data processing, ML, and notebooks (PySpark, Scala, .NET, SQL).
🔀
Pipelines
Data Factory-style orchestration engine built directly into the workspace.
🔗
Synapse Link
Near real-time replication from Cosmos DB / Dataverse / SQL, no ETL required.
💡
Synapse Studio is the single web UI for all of this — notebooks, SQL scripts, pipelines, and monitoring live side by side in one workspace.
02

A Synapse workspace is the top-level Azure resource. It is linked to a primary Azure Data Lake Storage Gen2 account/container ("primary storage") which acts as the default file system for Spark, serverless SQL, and pipeline staging.

03

Typical End-to-End Workflow

  • Ingest — Pipelines/Dataflows pull data from 90+ connectors into the lake or a staging table.
  • Transform — Spark notebooks or Mapping Data Flows clean & reshape the data (bronze → silver → gold).
  • Load / Model — Curated data lands in dedicated SQL pool tables or stays as Parquet queried via serverless SQL.
  • Serve — Power BI (Direct Lake / Import / DirectQuery), reporting tools, or downstream apps consume the results.
  • Monitor — Synapse Studio's Monitor hub tracks pipeline runs, Spark jobs, and SQL requests.

Synapse SQL

Two flavors of T-SQL compute in one workspace: provisioned, high-performance dedicated SQL pools for enterprise warehousing, and on-demand serverless SQL pools for ad-hoc queries directly over lake files.

Dedicated Pool Serverless Pool MPP
01

Dedicated SQL Pool

A dedicated SQL pool (formerly SQL DW) is a provisioned, massively parallel processing (MPP) data warehouse. Compute is billed continuously in DWUs (Data Warehouse Units) whether or not queries are running — you pause it to stop billing.

ConceptMeaning
Control nodeReceives queries, builds the distributed execution plan
Compute nodesExecute the plan in parallel; count scales with DWU tier
Distributions60 physical storage/compute units data is spread across
DWUBilling unit — higher DWU = more compute nodes = faster, pricier

Table distribution strategies

t-sql — create table with distribution
-- HASH distribution — best for large fact tables joined on a key
CREATE TABLE dbo.FactSales (
  SaleId INT, ProductId INT, Amount DECIMAL(10,2)
)
WITH (
  DISTRIBUTION = HASH(ProductId),
  CLUSTERED COLUMNSTORE INDEX
);

-- ROUND_ROBIN — even spread, no join key affinity, good for staging
CREATE TABLE stg.RawLoad (...) WITH (DISTRIBUTION = ROUND_ROBIN);

-- REPLICATE — small dimension tables copied to every node
CREATE TABLE dbo.DimDate (...) WITH (DISTRIBUTION = REPLICATE);
⚠️
A poor distribution key choice causes data skew — one node ends up with far more rows than others, slowing every query. Choose a high-cardinality column used in joins/group-bys.

Loading data — COPY & PolyBase

t-sql — COPY INTO
COPY INTO dbo.FactSales
FROM 'https://mystorage.blob.core.windows.net/data/sales/*.parquet'
WITH (
  FILE_TYPE = 'PARQUET',
  CREDENTIAL = (IDENTITY = 'Managed Identity')
);

COPY INTO is the modern, simplest bulk-load statement; PolyBase (external tables) is the older pattern still used for querying external data in place.

02

Serverless SQL Pool

A serverless SQL pool is always available per workspace — no provisioning, no pausing. You pay only for the data scanned per query (per TB), making it ideal for exploring files in the lake without loading them anywhere.

t-sql — query lake files directly
SELECT TOP 100 *
FROM OPENROWSET(
    BULK 'https://mystorage.dfs.core.windows.net/data/sales/*.parquet',
    FORMAT = 'PARQUET'
) AS rows;

-- Create a reusable view over the files
CREATE VIEW vw_sales AS
SELECT *
FROM OPENROWSET(
    BULK 'https://mystorage.dfs.core.windows.net/data/sales/',
    FORMAT = 'PARQUET'
) AS r;

03

CETAS & External Tables

CREATE EXTERNAL TABLE AS SELECT (CETAS) writes query results back out to the lake as Parquet/CSV — used to persist transformed data for reuse by other engines.

t-sql — CETAS
CREATE EXTERNAL DATA SOURCE lake_ds
WITH (LOCATION = 'https://mystorage.dfs.core.windows.net/data');

CREATE EXTERNAL FILE FORMAT parquet_fmt WITH (FORMAT_TYPE = PARQUET);

CREATE EXTERNAL TABLE ext.SalesSummary
WITH (
  LOCATION = '/curated/sales_summary/',
  DATA_SOURCE = lake_ds,
  FILE_FORMAT = parquet_fmt
)
AS
SELECT ProductId, SUM(Amount) AS Total
FROM dbo.FactSales
GROUP BY ProductId;

Apache Spark Pools

Managed, auto-scaling Apache Spark clusters inside the Synapse workspace for large-scale data engineering, exploratory analysis and machine learning using notebooks.

Auto-scale Auto-pause Notebooks
01

Spark Pool Basics

A Spark pool is a set of configuration (node size, min/max nodes, Spark & Python versions) — the actual cluster only spins up when a job is submitted, and shuts down automatically after an idle timeout.

SettingDescription
Node sizeSmall / Medium / Large / XLarge / XXLarge — vCores & memory per node
AutoscaleMin/max node count; Spark scales nodes up/down with workload
Auto-pauseCluster terminates after N idle minutes to stop billing
LanguagesPySpark (Python), Spark SQL, Scala, .NET (C#/F#)
02

Notebooks & DataFrames

pyspark — read, transform, write
# Read from the primary ADLS Gen2 lake
df = spark.read.option("header", True) \
    .csv("abfss://data@mystorage.dfs.core.windows.net/raw/sales/")

# Transform
clean = df.filter(df.Amount > 0).dropDuplicates()

# Write back as Parquet, partitioned
clean.write.mode("overwrite") \
    .partitionBy("Region") \
    .parquet("abfss://data@mystorage.dfs.core.windows.net/curated/sales/")

# Register as a Spark SQL temp view for ad-hoc querying
clean.createOrReplaceTempView("sales")
spark.sql("SELECT Region, SUM(Amount) FROM sales GROUP BY Region").show()
📘
Notebooks support mixed-language cells — start a cell with %%sql, %%pyspark, %%spark (Scala), or %%csharp to switch languages within one notebook.
03

Talking to SQL Pools from Spark

pyspark — write to dedicated sql pool
%%pyspark
clean.write \
    .format("com.databricks.spark.sqldw") \
    .option("url", jdbcUrl) \
    .option("tempDir", "abfss://staging@mystorage.dfs.core.windows.net/tmp") \
    .option("forwardSparkAzureStorageCredentials", "true") \
    .option("dbTable", "dbo.FactSales") \
    .mode("append").save()

Synapse's built-in Spark-to-SQL connector stages data through the lake automatically, avoiding slow row-by-row JDBC writes.

02

Copy Activity & Data Flows

The Copy activity moves data between a source and sink with optional schema mapping, using scale-out integration runtimes for throughput. Mapping Data Flows add a visual, no-code transformation layer (joins, aggregates, pivots) that compiles down to Spark under the hood.

json — pipeline definition (simplified)
{
  "name": "CopySalesToLake",
  "activities": [
    {
      "name": "CopyFromSql",
      "type": "Copy",
      "source": { "type": "AzureSqlSource" },
      "sink": { "type": "ParquetSink" }
    },
    {
      "name": "RunNotebook",
      "type": "SynapseNotebook",
      "dependsOn": [{ "activity": "CopyFromSql" }]
    }
  ]
}
💡
Use dependsOn conditions (Succeeded, Failed, Completed) to chain activities and build branching, fault-tolerant pipelines.
03

Triggers

  • Schedule trigger — runs on a wall-clock recurrence (e.g. every day at 2 AM).
  • Tumbling window trigger — fixed, non-overlapping time windows with built-in retry/backfill for late data.
  • Event-based trigger — fires on blob created/deleted events in a storage account.
  • Manual trigger — run on-demand via Studio, REST API, or SDK.

Workspace-level access control, data protection features, and near real-time replication from operational stores straight into Synapse for analytics — no traditional ETL pipeline required.

t-sql — RLS & masking
-- Dynamic Data Masking
ALTER TABLE dbo.Customers
ALTER COLUMN Email ADD MASKED WITH (FUNCTION = 'email()');

-- Row-Level Security predicate + policy
CREATE FUNCTION sec.fn_predicate(@Region sysname)
RETURNS TABLE WITH SCHEMABINDING AS RETURN SELECT 1 AS ok
WHERE @Region = USER_NAME();

CREATE SECURITY POLICY RegionPolicy
ADD FILTER PREDICATE sec.fn_predicate(Region) ON dbo.Sales;
02

Azure Synapse Link

Synapse Link continuously replicates operational data into Synapse in near real-time, in an analytics-optimized columnar format, without impacting the source system's performance (no traditional ETL job needed).

🌍
Cosmos DB
HTAP pattern — analytical store auto-syncs from the transactional store.
📇
Dataverse
Power Platform / Dynamics 365 data replicated into the lake as CDM folders.
🗃️
Azure SQL DB / SQL Server
Change feed-based replication for near real-time analytics on operational data.
📘
Because the analytical store is separate from the transactional store, heavy analytics workloads never compete with production application traffic (HTAP = Hybrid Transactional/Analytical Processing).
03

Monitoring

  • Monitor hub — pipeline runs, trigger runs, Spark application history, SQL request history, all in one place.
  • Dynamic Management Views (DMVs) — real-time visibility into dedicated SQL pool query execution & resource usage.
  • Azure Monitor / Log Analytics — diagnostic logs and metrics exported for alerting & long-term retention.
