What is Apache Spark?

Apache Spark is an open-source, distributed computing engine for large-scale data processing. It was created at UC Berkeley in 2009 and open-sourced in 2010. PySpark is the Python API for Apache Spark.

  • In-memory computing — up to 100x faster than Hadoop MapReduce for iterative algorithms
  • Unified engine — batch, streaming, ML, and graph processing in one system
  • Fault tolerant — automatically recovers lost partitions using lineage
  • Lazy evaluation — transformations build a DAG; execution happens only on actions
💡 On Databricks, a Spark cluster is already configured. You just need to create a SparkSession to connect to it.
In [2]
MARKDOWN

Spark Architecture

Components

  • Driver — the main process that runs user code, creates SparkContext, schedules tasks
  • Executor — JVM process on worker nodes that runs tasks and stores data
  • Cluster Manager — allocates resources (YARN, Mesos, Standalone, Kubernetes)
  • SparkContext — entry point to Spark cluster (low-level API)
  • SparkSession — unified entry point (wraps SparkContext, SQLContext, HiveContext)

Execution Model

  • Job — triggered by an action (count, show, write)
  • Stage — set of tasks separated by shuffle boundaries
  • Task — unit of work on a single partition
  • DAG — Directed Acyclic Graph of transformations

Lazy Evaluation — Core Concept

Spark does not execute transformations immediately. It builds a DAG (plan) and only executes when an action is called.

⚠️ Transformations (map, filter, select, groupBy) → lazy, return new RDD/DF
⚡ Actions (count, show, collect, write) → trigger execution
In [6]
PYTHON
# Lazy evaluation demo import time rdd = spark.sparkContext.parallelize(range(1_000_000)) # This does NOT execute yet — just builds the DAG filtered = rdd.filter(lambda x: x % 2 == 0) doubled = filtered.map(lambda x: x * 2) print("Plan built — no computation yet") # THIS triggers execution (action) t0 = time.time() result = doubled.count() # ← action fires the DAG print(f"Count: {result} (took {time.time()-t0:.2f}s)")

What is an RDD?

An RDD is the fundamental data structure of Spark — an immutable, distributed collection of objects partitioned across the cluster.

  • Resilient — fault-tolerant via lineage (can recompute lost partitions)
  • Distributed — data is split into partitions across nodes
  • Dataset — a collection of any Python objects
⚠️ RDDs are lower-level. Prefer DataFrames for tabular data — they use Catalyst optimizer and are much faster. Use RDDs for unstructured data or when you need fine-grained control.

DataFrames in PySpark

A DataFrame is a distributed collection of data organized into named columns — similar to a SQL table or pandas DataFrame, but distributed across the cluster.

  • Built on top of RDDs but use the Catalyst optimizer for query optimization
  • Support schema enforcement (typed columns)
  • Can be queried with Python API or SQL
  • Immutable — transformations return new DataFrames
💡 DataFrames are 10–100x faster than RDDs for tabular data because of the Catalyst optimizer and Tungsten execution engine.

Transformations vs Actions

Every DataFrame/RDD operation is either a transformation (lazy — builds the DAG) or an action (eager — triggers execution).

  • Narrow transformation — each input partition maps to exactly one output partition. No shuffle. (e.g., filter, map, select, withColumn)
  • Wide transformation — requires data from multiple partitions, causes a shuffle. (e.g., groupBy, join, distinct, orderBy)
⚠️ Wide transformations are expensive — they move data across the network. Minimize shuffles for performance.

Join Performance Tips

  • Broadcast Join — if one side is small (<10MB), broadcast it to all executors to avoid shuffle
  • Sort-Merge Join — default for large-large joins; sorts and merges both sides
  • Shuffle Hash Join — used when one side fits in memory per partition
  • Filter before join — always filter rows before joining to reduce data
  • Avoid cross joins — result is N×M rows, extremely expensive
⚠️ Avoid calling collect() on large DataFrames after a join. Use show(), limit(), or write to storage.

What is Delta Lake?

Delta Lake is an open-source storage layer bringing ACID transactions to Spark and big data workloads. Every write creates a new version recorded in _delta_log/.

  • ACID transactions — atomic commits, safe concurrent reads and writes
  • Schema enforcement + evolution — rejects bad writes; can add columns safely
  • Time travel — query any historical version by number or timestamp
  • MERGE (upsert) — insert, update, and delete in one atomic operation
  • Z-order clustering — co-locate related data for fast filter queries
  • Change Data Feed (CDF) — track row-level changes for CDC pipelines
  • Streaming + batch unified — same table serves both

When to Cache

  • ✅ DataFrame is used multiple times (multiple actions or reused in a loop)
  • ✅ Expensive computation (complex joins, many aggregations)
  • ✅ Iterative ML algorithms — same data accessed many times
  • ❌ DataFrame is used only once — caching adds overhead
  • ❌ Very large data that won't fit in cluster memory
  • ❌ Streaming DataFrames
⚠️ Caching is lazy — the data is stored only after the first action. Always trigger an action (count, show) after cache() to materialize it immediately.
