03

Joins

t-sql — join types
SELECT e.Name, d.DeptName
FROM Employees e
INNER JOIN Departments d ON e.DeptId = d.DeptId;    -- only matching rows

SELECT e.Name, d.DeptName
FROM Employees e
LEFT JOIN Departments d ON e.DeptId = d.DeptId;     -- all employees, matched dept or NULL

SELECT e.Name, d.DeptName
FROM Employees e
FULL OUTER JOIN Departments d ON e.DeptId = d.DeptId; -- everything, matched or not

SELECT e1.Name AS Emp, e2.Name AS Manager
FROM Employees e1
JOIN Employees e2 ON e1.ManagerId = e2.EmpId;         -- self join
JoinReturns
INNER JOINOnly rows with matches on both sides
LEFT JOINAll left rows + matched right rows (NULL if none)
RIGHT JOINAll right rows + matched left rows (NULL if none)
FULL OUTER JOINAll rows from both sides, matched where possible
CROSS JOINCartesian product — every row × every row
SELF JOINA table joined to itself (e.g. employee ↔ manager)
04

Aggregation & Grouping

t-sql — group by & having
SELECT DeptId, COUNT(*) AS Headcount, AVG(Salary) AS AvgSalary
FROM Employees
GROUP BY DeptId
HAVING COUNT(*) > 5          -- filters groups, WHERE filters rows
ORDER BY AvgSalary DESC;
📘
Logical query processing order: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY. WHERE can't reference column aliases defined in SELECT because it runs first.
05

Subqueries & CTEs

t-sql — subqueries & CTE
-- Correlated subquery
SELECT Name, Salary
FROM Employees e
WHERE Salary > (SELECT AVG(Salary) FROM Employees WHERE DeptId = e.DeptId);

-- Common Table Expression (CTE)
WITH DeptAvg AS (
  SELECT DeptId, AVG(Salary) AS AvgSal FROM Employees GROUP BY DeptId
)
SELECT e.Name, e.Salary, d.AvgSal
FROM Employees e
JOIN DeptAvg d ON e.DeptId = d.DeptId;

-- Recursive CTE — org chart traversal
WITH OrgChart AS (
  SELECT EmpId, Name, ManagerId, 0 AS Lvl FROM Employees WHERE ManagerId IS NULL
  UNION ALL
  SELECT e.EmpId, e.Name, e.ManagerId, o.Lvl + 1
  FROM Employees e JOIN OrgChart o ON e.ManagerId = o.EmpId
)
SELECT * FROM OrgChart;
06

Window Functions

t-sql — window functions
SELECT Name, DeptId, Salary,
  ROW_NUMBER() OVER (PARTITION BY DeptId ORDER BY Salary DESC) AS rn,
  RANK()       OVER (PARTITION BY DeptId ORDER BY Salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY DeptId ORDER BY Salary DESC) AS drnk,
  LAG(Salary,1)  OVER (PARTITION BY DeptId ORDER BY Salary) AS PrevSalary,
  SUM(Salary)   OVER (PARTITION BY DeptId) AS DeptTotal
FROM Employees;
FunctionBehavior
ROW_NUMBER()Unique sequential number, no ties
RANK()Same rank for ties, gaps after ties (1,2,2,4)
DENSE_RANK()Same rank for ties, no gaps (1,2,2,3)
LAG / LEADValue from a previous / next row in the window
SUM/AVG OVER()Running or partitioned aggregate without collapsing rows

03

High Availability & Disaster Recovery

  • Built-in HA — every Azure SQL tier includes automatic replication & failover within a region (architecture depends on tier).
  • Active geo-replication — up to 4 readable secondary databases in other regions, manual failover.
  • Auto-failover groups — group of databases that fail over together to a secondary region, with a listener endpoint so connection strings don't change.
  • Point-in-time restore (PITR) — restore to any point within the retention window (7–35 days) automatically, no extra config.
  • Long-term retention (LTR) — weekly/monthly/yearly backups kept up to 10 years.

Stored Procedures & Functions

t-sql — proc & scalar/table functions
CREATE PROCEDURE usp_GetEmployeesByDept
  @DeptId INT
AS
BEGIN
  SET NOCOUNT ON;
  SELECT EmpId, Name, Salary FROM Employees WHERE DeptId = @DeptId;
END;
EXEC usp_GetEmployeesByDept @DeptId = 2;

-- Scalar function
CREATE FUNCTION fn_AnnualBonus(@Salary DECIMAL(10,2))
RETURNS DECIMAL(10,2)
AS BEGIN RETURN @Salary * 0.1; END;

-- Inline table-valued function
CREATE FUNCTION fn_TopEarners(@DeptId INT)
RETURNS TABLE AS RETURN
  SELECT Name, Salary FROM Employees WHERE DeptId = @DeptId AND Salary > 80000;
📘
Stored procedures can perform DML and have side effects; functions must be deterministic-ish and cannot modify database state (no INSERT/UPDATE/DELETE on permanent tables).

  UPDATE Accounts SET Balance = Balance + 100 WHERE AcctId = 2;

END TRY

END CATCH;

Triggers & Views

t-sql — trigger & view
CREATE TRIGGER trg_AuditSalaryChange
ON Employees
AFTER UPDATE AS
BEGIN
  INSERT INTO SalaryAudit (EmpId, OldSalary, NewSalary, ChangedAt)
  SELECT i.EmpId, d.Salary, i.Salary, GETDATE()
  FROM inserted i JOIN deleted d ON i.EmpId = d.EmpId
  WHERE i.Salary <> d.Salary;
END;

CREATE VIEW vw_ActiveEmployees AS
SELECT EmpId, Name, DeptId FROM Employees WHERE IsActive = 1;
💡
Inside a trigger, the special inserted and deleted tables hold the new and old row images — used heavily for auditing patterns.

Indexes

Index typeDescription
ClusteredDefines the physical row order; one per table (often the PK)
NonclusteredSeparate structure pointing back to the data row; many per table

Covering indexIncludes all columns a query needs, avoiding a lookup back to the base table

t-sql — creating indexes
CREATE NONCLUSTERED INDEX IX_Employees_DeptId
ON Employees(DeptId)
INCLUDE (Name, Salary);          -- covering index

CREATE UNIQUE INDEX UX_Employees_Email ON Employees(Email);

CREATE NONCLUSTERED INDEX IX_ActiveOnly
ON Employees(DeptId) WHERE IsActive = 1;   -- filtered index
⚠️
Every index speeds up reads but slows down writes (INSERT/UPDATE/DELETE must maintain it) and adds storage — index only what your query patterns actually need.
02

Execution Plans

The query optimizer picks an execution plan based on statistics (row count estimates). Reading the plan reveals what's actually slow.

OperatorMeaning
Table Scan / Clustered Index ScanReads every row — expensive on large tables, often means a missing index
Index SeekJumps directly to matching rows via an index — fast
Key LookupNonclustered index found the row, then fetches remaining columns from the clustered index
Nested Loops JoinGood for small outer input joined to an indexed inner input
Hash Match JoinGood for large, unsorted inputs without useful indexes
SortExplicit sort operator — can be costly; sometimes avoidable with the right index order
t-sql — inspect a plan
SET STATISTICS IO ON;
SET STATISTICS TIME ON;
SELECT * FROM Employees WHERE DeptId = 2;
-- or in SSMS: Ctrl+M to include the actual execution plan
03

Tuning Tools

  • Query Store — tracks query plans & runtime stats over time, lets you force a known-good plan.
  • Database Engine Tuning Advisor — analyzes a workload and recommends indexes/partitioning.
  • Automatic tuning (Azure SQL) — can auto-create/drop indexes and auto-correct plan regressions.
  • sys.dm_exec_query_stats / sys.dm_db_index_usage_stats — DMVs for finding expensive or unused indexes.
  • UPDATE STATISTICS — refreshes the optimizer's row-count estimates when data has changed significantly.
Page 05 / Interview Preparation

SQL Interview Prep

Common SQL / Azure SQL interview questions, grouped by topic. Click a question to reveal a concise answer you can build on out loud.

Conceptual Coding Azure-specific
01

Core SQL Concepts

What's the difference between DELETE, TRUNCATE, and DROP?▸
DELETE removes rows one at a time, is logged, can be filtered with WHERE, and can be rolled back. TRUNCATE deallocates all pages at once, is minimally logged, resets identity, and can't use WHERE. DROP removes the entire table object, including its structure, from the database.
What is the difference between WHERE and HAVING?▸
WHERE filters individual rows before grouping; HAVING filters groups after GROUP BY has aggregated them. You can't use an aggregate function like SUM() in WHERE, but you can in HAVING.
Explain the difference between UNION and UNION ALL.▸
UNION combines result sets and removes duplicate rows (requires an implicit sort/dedupe, so it's slower). UNION ALL keeps all rows including duplicates and is faster since no dedupe pass is needed.
What is normalization? Explain 1NF, 2NF, 3NF briefly.▸
Normalization organizes data to reduce redundancy and avoid update anomalies. 1NF: atomic columns, no repeating groups. 2NF: 1NF + every non-key column depends on the whole primary key (no partial dependency on composite keys). 3NF: 2NF + no column depends on another non-key column (no transitive dependency).
What's the difference between a primary key and a unique key?▸
A table can have only one primary key, which disallows NULLs and is typically the clustering key by default. A table can have multiple unique keys/constraints, and a unique key can allow one NULL value (per column, in SQL Server).
What is a NULL and how does it behave in comparisons?▸
NULL represents unknown/missing data. Any comparison with NULL using = or <> returns UNKNOWN, not TRUE/FALSE — that's why you must use IS NULL / IS NOT NULL. Aggregate functions (except COUNT(*)) ignore NULLs.
02

Joins, Subqueries & Set Operations

What's the difference between a correlated and a non-correlated subquery?▸
A non-correlated subquery runs once, independently of the outer query. A correlated subquery references a column from the outer query and conceptually re-executes for each outer row — generally slower unless the optimizer rewrites it as a join.
How would you find duplicate rows in a table?▸
Group by the columns that define a "duplicate" and filter with HAVING COUNT(*) > 1, e.g. SELECT Email, COUNT(*) FROM Users GROUP BY Email HAVING COUNT(*) > 1. To delete duplicates while keeping one, use ROW_NUMBER() in a CTE and delete rows where rn > 1.
What is the difference between EXISTS and IN?▸
EXISTS stops as soon as it finds one matching row (semi-join, efficient with large/correlated subqueries) and treats NULLs safely. IN materializes the full list and can behave unexpectedly if that list contains NULL (with NOT IN, a NULL in the list makes the whole condition return no rows).
How do you find the second-highest salary in a table?▸
Several valid approaches: SELECT MAX(Salary) FROM Employees WHERE Salary < (SELECT MAX(Salary) FROM Employees), or using DENSE_RANK() OVER (ORDER BY Salary DESC) in a CTE and filtering rank = 2 — the window-function approach also generalizes cleanly to "Nth highest."
03

Indexing & Performance

What's the difference between a clustered and non-clustered index?▸
A clustered index determines the physical storage order of the table's rows — there can only be one. A non-clustered index is a separate structure with a copy of the key columns plus a pointer (or clustering key) back to the actual row — a table can have many.
What is a covering index and why does it help?▸
A covering index includes every column a query needs (via key columns and/or INCLUDE columns), so SQL Server can satisfy the query entirely from the index without a "key lookup" back to the base table — this reduces I/O significantly.
Why might an index not be used even though it exists?▸
Common causes: outdated statistics, a function or implicit conversion applied to the indexed column ("non-sargable" predicate, e.g. WHERE YEAR(OrderDate)=2026), low selectivity (the optimizer decides a scan is cheaper), leading wildcard LIKE searches (LIKE '%abc'), or parameter sniffing producing a bad cached plan.
What is parameter sniffing?▸
SQL Server compiles and caches a query plan based on the parameter values used the first time it runs. If later calls use very different values with a different optimal plan, performance can suffer. Fixes include OPTION (RECOMPILE), query hints, or plan guides/Query Store forced plans.
04

Transactions & Concurrency

Explain ACID properties.▸
Atomicity: all statements in a transaction succeed or none do. Consistency: a transaction moves the database from one valid state to another, respecting constraints. Isolation: concurrent transactions don't see each other's uncommitted changes (level depends on isolation setting). Durability: once committed, changes survive a crash.
What is a deadlock and how do you resolve/avoid it?▸
A deadlock happens when two transactions each hold a lock the other needs, so neither can proceed. SQL Server detects this and kills one transaction (the "deadlock victim"). Avoid it by accessing tables/rows in a consistent order across transactions, keeping transactions short, and using appropriate isolation levels.
What's the difference between optimistic and pessimistic concurrency?▸
Pessimistic concurrency locks data as soon as it's read, blocking others until the transaction finishes. Optimistic concurrency (e.g. SNAPSHOT isolation, row versioning) lets transactions read without locking and only checks for conflicts at commit time — better throughput when conflicts are rare.
05

Azure SQL Specific

When would you pick Azure SQL Managed Instance over Azure SQL Database?▸
When you need near-complete SQL Server instance-level surface area — cross-database queries/transactions, SQL Server Agent, CLR, Service Broker, or a straightforward lift-and-shift of an on-prem instance with minimal app changes — while still wanting PaaS management.
What's the difference between DTU-based and vCore-based purchasing models?▸
DTU bundles compute, storage, and I/O into one abstract unit across a few fixed tiers — simple but less transparent. vCore lets you choose compute and storage independently, supports Azure Hybrid Benefit (bring your own SQL Server license for savings), and includes serverless/Hyperscale options DTU doesn't.
What is an auto-failover group used for?▸
It groups one or more databases that fail over together to a secondary region, and exposes read-write and read-only listener endpoints so the application's connection string doesn't need to change after a failover — simplifying disaster recovery for multi-database applications.
How does Azure SQL Database handle backups by default?▸
Automatic backups (full weekly, differential every ~12 hours, log backups every 5–10 minutes) are taken with no configuration needed, enabling point-in-time restore within a 7–35 day retention window; long-term retention policies can keep backups for up to 10 years.
06

Scenario / Coding Questions

Write a query to find employees earning more than their manager.▸
SELECT e.Name FROM Employees e JOIN Employees m ON e.ManagerId = m.EmpId WHERE e.Salary > m.Salary; — a classic self-join question testing whether you can relate a table to itself.
Write a query to get the top 3 highest-paid employees per department.▸
Use a window function in a CTE: WITH Ranked AS (SELECT *, DENSE_RANK() OVER (PARTITION BY DeptId ORDER BY Salary DESC) AS rnk FROM Employees) SELECT * FROM Ranked WHERE rnk <= 3; — this "top N per group" pattern comes up constantly.
How would you pivot rows into columns in T-SQL?▸
Use the PIVOT operator, or a manual conditional aggregate: SELECT DeptId, SUM(CASE WHEN Year=2025 THEN Amount END) AS Y2025, SUM(CASE WHEN Year=2026 THEN Amount END) AS Y2026 FROM Sales GROUP BY DeptId; — the CASE approach is often easier to reason about and extend.
How do you safely upsert data (insert-or-update) in T-SQL?▸
Use MERGE with WHEN MATCHED / WHEN NOT MATCHED clauses, wrapped in an explicit transaction with the right isolation level (or a HOLDLOCK hint) to avoid race conditions under concurrency — a plain "check-then-insert" without locking can duplicate rows under load.
