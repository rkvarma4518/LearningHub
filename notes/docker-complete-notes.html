Docker Fundamentals

Docker packages an application with everything it needs to run — code, runtime, libraries, system tools — into a portable, isolated unit called a container, so it behaves identically on any machine.

Containers Images Linux Namespaces cgroups
01

What Is Docker & Why Containers?

Docker is a platform for building, shipping, and running applications inside lightweight, isolated units called containers. Unlike a virtual machine, a container shares the host's OS kernel instead of virtualizing hardware — making it far faster to start and much lighter on resources.

AspectVirtual MachineDocker Container
Isolation unitFull guest OS + kernelProcess, isolated via kernel namespaces
Boot timeMinutesMilliseconds to seconds
SizeGBs (full OS)MBs (app + minimal deps)
Density per hostTensHundreds to thousands
Hypervisor neededYes (Type 1/2)No — uses host kernel directly
💡
"Build once, run anywhere" — a container that runs on a developer's laptop runs identically in CI, staging, and production because the image bundles the entire runtime environment.
02

Docker uses a client-server model. The Docker CLI talks to the Docker daemon (dockerd) over a REST API (usually via a Unix socket), which does the actual work of building, running, and managing containers.

03

Underlying Linux Technology

  • Namespaces — isolate what a process can see: PID (process IDs), NET (network stack), MNT (filesystem mounts), UTS (hostname), IPC (inter-process comms), USER (user/group IDs).
  • cgroups (control groups) — limit and account for resource usage (CPU, memory, I/O, network) per container, preventing one container from starving the host.
  • Union filesystem (OverlayFS) — stacks image layers into a single merged view; a thin writable layer sits on top for the running container.
  • OCI (Open Container Initiative) — the industry-standard spec for image format and runtime that Docker, containerd, and Kubernetes all implement, ensuring interoperability.
📘
A container is fundamentally just a regular Linux process — docker run doesn't create a new kind of thing, it launches a process with restricted namespaces and cgroup limits applied.

Page 02 / Docker

Images & Dockerfile

Every container starts life as an image — a stack of read-only layers built from a Dockerfile's instructions, cached and reused for speed.

Layers Dockerfile Multi-stage Builds
01

Image Layers & Caching

Each instruction in a Dockerfile that changes the filesystem (RUN, COPY, ADD) creates a new, immutable layer. Layers are cached and content-addressed — if an instruction and its inputs haven't changed, Docker reuses the cached layer instead of rebuilding it.

💡
Order Dockerfile instructions from least to most frequently changing (e.g. install dependencies before copying source code) so cache hits happen more often and builds stay fast.
02

Dockerfile Instructions

dockerfile — annotated example
FROM node:20-alpine                 # base image
LABEL maintainer="team@example.com"
ARG BUILD_ENV=production            # build-time only variable
ENV NODE_ENV=$BUILD_ENV              # runtime environment variable
WORKDIR /app                        # sets cwd for following instructions
COPY package*.json ./           # copy only manifest first (cache-friendly)
RUN npm ci --only=production
COPY . .                            # copy the rest of the source
EXPOSE 3000                         # documents the listening port
USER node                           # drop root privileges
HEALTHCHECK --interval=30s CMD wget -q --spider http://localhost:3000/health || exit 1
ENTRYPOINT ["node"]                  # fixed executable
CMD ["server.js"]                  # default argument(s), overridable
InstructionPurpose
FROMSets the base image; every Dockerfile starts here (or FROM scratch)
RUNExecutes a command at build time, creating a new layer
COPYCopies files from build context into the image
ADDLike COPY, but also auto-extracts tar archives and fetches remote URLs
WORKDIRSets the working directory for subsequent instructions
ENVSets a persistent environment variable, available at runtime
ARGBuild-time-only variable, not present in the final container
EXPOSEDocuments which port the container listens on (doesn't publish it)
VOLUMEDeclares a mount point for persistent/external storage
USERSets the user (and group) subsequent instructions/container run as
CMDDefault command/args for the container; overridden by `docker run` args
ENTRYPOINTFixed executable the container always runs; CMD supplies default args to it
HEALTHCHECKCommand Docker runs periodically to report container health status
⚠️
CMD vs ENTRYPOINT is a classic gotcha: with only CMD, `docker run image echo hi` replaces the whole command. With ENTRYPOINT set, extra `docker run` arguments are appended to it instead of replacing it.
03

Building Images

bash — build & inspect
docker build -t myapp:1.0 .                 # build from Dockerfile in current dir
docker build -f Dockerfile.prod -t myapp:prod .
docker history myapp:1.0                    # see each layer and its size
docker image inspect myapp:1.0
docker images                               # list local images
docker image prune -a                     # remove unused images

.dockerignore

Just like .gitignore, this file excludes files (node_modules, .git, secrets, local env files) from the build context sent to the daemon — smaller context, faster builds, and no accidental secret leakage into layers.

04

Multi-Stage Builds

Multi-stage builds use multiple FROM statements in one Dockerfile — earlier stages compile/build, and the final stage copies only the needed artifacts, keeping the final image small and free of build tools.

dockerfile — multi-stage build
FROM golang:1.22 AS builder
WORKDIR /src
COPY . .
RUN go build -o /app .

FROM gcr.io/distroless/base-debian12
COPY --from=builder /app /app
USER nonroot:nonroot
ENTRYPOINT ["/app"]
💡
Using a minimal or distroless final-stage base image shrinks the attack surface — no shell, no package manager, no extra binaries an attacker could use.
05

docker login myregistry.azurecr.io
docker push myregistry.azurecr.io/myapp:1.0
docker pull myregistry.azurecr.io/myapp:1.0

bash — volumes & mounts
docker volume create db-data
docker run -d -v db-data:/var/lib/postgresql/data postgres:16
docker run -d -v $(pwd):/app -w /app node:20 npm run dev   # bind mount
docker run -d --tmpfs /app/cache nginx                      # tmpfs mount
docker volume ls
docker volume prune
⚠️
Containers are ephemeral by default — anything written to the container's own writable layer is lost with docker rm. Always use a volume or bind mount for data that must survive.

Compose & Orchestration

Defining and running multi-container applications declaratively, and understanding where Docker's own orchestration ends and Kubernetes begins.

docker-compose.yml Swarm Best Practices
01

Docker Compose

Docker Compose defines a multi-container application in a single YAML file — services, networks, volumes, env vars — and brings the whole stack up or down with one command.

yaml — docker-compose.yml
services:
  api:
    build: .
    ports:
      - "5000:5000"
    environment:
      - DB_HOST=db
    depends_on:
      - db
    restart: unless-stopped

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=${DB_PASSWORD}

volumes:
  db-data:
bash — compose commands
docker compose up -d              # build + start everything, detached
docker compose ps
docker compose logs -f api
docker compose exec api sh
docker compose down -v              # stop & remove containers, networks, volumes
docker compose up -d --scale api=3     # run 3 replicas of a service
📘
Compose is ideal for local development and simple single-host deployments. For multi-host, self-healing, auto-scaling production workloads, teams move to Swarm or Kubernetes.
02

03

Dockerfile & Image Best Practices

  • Use small, official base images — alpine or distroless variants instead of full OS images.
  • Multi-stage builds to exclude compilers/build tools from the final image.
  • Don't run as root — set a non-root USER in the Dockerfile.
  • One process per container — keep containers single-purpose, composable, and easy to scale independently.
  • Never bake secrets into the image — use build secrets (--secret), env vars, or a secrets manager instead of ARG/ENV for credentials.
  • Pin versions — avoid `latest`; pin base image tags/digests for reproducible builds.
  • Minimize layers — combine related RUN commands with && and clean up caches in the same layer.
  • Use .dockerignore to keep the build context small and avoid leaking files.

What Is Trivy?

Trivy (by Aqua Security) is a fast, open-source, all-in-one scanner. Besides container images, it scans filesystems, git repositories, Kubernetes clusters, and Infrastructure-as-Code files for vulnerabilities, misconfigurations, secrets, and license issues — and can output a full SBOM.

bash — installing trivy
# macOS (Homebrew)
brew install aquasecurity/trivy/trivy

# Debian/Ubuntu (apt repo)
sudo apt-get install wget apt-transport-https gnupg
wget -qO - https://aquasecurity.github.io/trivy-repo/deb/public.key | sudo apt-key add -
echo "deb https://aquasecurity.github.io/trivy-repo/deb $(lsb_release -sc) main" | sudo tee -a /etc/apt/sources.list.d/trivy.list
sudo apt-get update && sudo apt-get install trivy

# Run via Docker itself (no local install needed)
docker run --rm -v /var/run/docker.sock:/var/run/docker.sock \
  aquasec/trivy image nginx:latest

# Binary install script
curl -sfL https://raw.githubusercontent.com/aquasecurity/trivy/main/contrib/install.sh | sh -s -- -b /usr/local/bin
04

Trivy — Online (Default) Scanning

By default Trivy connects to the internet to download its vulnerability database (built from NVD, GitHub Security Advisories, distro security trackers) into a local cache before scanning — this is the normal mode for laptops, build servers, and CI runners with outbound internet access.

bash — scan an image (online mode)
# Scan a local or remote image
trivy image nginx:1.25

# Scan directly from a registry, without pulling to local docker
trivy image myregistry.azurecr.io/myapp:1.0

# Only show HIGH and CRITICAL severities
trivy image --severity HIGH,CRITICAL nginx:1.25

# Fail the command (exit code 1) if any CRITICAL vuln is found — great for CI gates
trivy image --severity CRITICAL --exit-code 1 myapp:1.0

# Scan a Dockerfile / repo / filesystem directly
trivy config ./Dockerfile
trivy fs .
trivy repo https://github.com/org/myrepo

# Generate a Software Bill of Materials (SBOM)
trivy image --format cyclonedx --output sbom.json myapp:1.0

# Scan for secrets baked into layers
trivy image --scanners secret myapp:1.0

# JSON output for programmatic consumption
trivy image --format json --output results.json myapp:1.0
💡
Trivy caches the DB locally (default ~$HOME/.cache/trivy) and refreshes it roughly every 6–12 hours by default — repeated scans on the same day don't always re-download it.
05

Trivy — Offline / Air-Gapped Scanning

In air-gapped environments (no outbound internet from the scanning host — common in secure enterprise/on-prem setups), you download the vulnerability DB on a machine that does have internet access, then transfer and use it offline.

Step 1 — Download the DB on an internet-connected machine

bash — fetch trivy-db & java-db as OCI artifacts
# Trivy publishes its vuln DB and Java DB as OCI artifacts on GHCR
oras pull ghcr.io/aquasecurity/trivy-db:2
oras pull ghcr.io/aquasecurity/trivy-java-db:1

# Alternative: let trivy itself download to a specific cache dir, then archive it
trivy image --download-db-only --cache-dir ./trivy-cache
tar czf trivy-db.tar.gz -C ./trivy-cache .

Step 2 — Transfer the archive to the air-gapped host

Move db.tar.gz / trivy-db.tar.gz via USB drive, internal file share, or an internal artifact repository — whatever your organization's approved offline transfer process is.

Step 3 — Scan offline

bash — offline scan using the pre-loaded db
# Extract the DB into trivy's expected cache location
mkdir -p ~/.cache/trivy/db
tar xzf trivy-db.tar.gz -C ~/.cache/trivy/db

# Tell trivy not to attempt any network calls
trivy image --skip-db-update --offline-scan myapp:1.0

# If using a private/internal OCI registry mirror instead of GHCR:
trivy image --db-repository myinternal.registry/trivy-db:2 myapp:1.0
FlagEffect
--skip-db-updateDon't attempt to check for or download a newer DB
--offline-scanNever make any network call during the scan (fully air-gapped mode)
--cache-dirCustom path to read/write the vulnerability DB cache
--db-repositoryPoint to an internal/private OCI mirror of the trivy-db instead of GHCR
⚠️
An offline DB goes stale — build a recurring process (weekly/monthly) to refresh and re-import the DB archive, or offline scans will miss newly disclosed CVEs.
06

CI/CD Integration

yaml — GitHub Actions example
- name: Build image
  run: docker build -t myapp:${{ github.sha }} .

- name: Scan with Trivy
  uses: aquasecurity/trivy-action@master
  with:
    image-ref: myapp:${{ github.sha }}
    severity: CRITICAL,HIGH
    exit-code: '1'
    format: sarif
    output: trivy-results.sarif

- name: Upload results to Security tab
  uses: github/codeql-action/upload-sarif@v3
  with:
    sarif_file: trivy-results.sarif
💡
Use exit-code: 1 paired with a severity filter to fail the pipeline — treat this as a merge/deploy gate, not just an informational report.
07

Additional Hardening Practices

  • Docker Bench for Security — CIS benchmark audit script checking host & daemon configuration.
  • Image signing — Docker Content Trust / Notary or Sigstore Cosign to verify an image hasn't been tampered with before deploy.
  • Least privilege — avoid --privileged, drop unneeded Linux capabilities (--cap-drop=ALL, add back only what's needed).
  • Read-only root filesystem — run containers with --read-only where the app doesn't need to write locally.
  • Never mount the Docker socket into untrusted containers — it's effectively root access to the host.
  • Regularly rebuild & rescan images even without code changes — new CVEs are discovered in base image packages constantly.

Docker Interview Prep

Common Docker interview questions grouped by topic. Click a question to reveal a concise answer you can build on out loud.

Conceptual Scenario Security
01

Core Concepts

What is the difference between a Docker image and a container?▸
An image is a read-only, layered template built from a Dockerfile — it's just files on disk. A container is a running (or stopped) instance of that image, with its own thin writable layer on top and its own isolated process/network/filesystem namespace.
What's the difference between a Docker container and a virtual machine?▸
A VM virtualizes hardware and runs a full guest OS with its own kernel via a hypervisor — heavier, slower to boot. A container shares the host kernel and is isolated using namespaces and cgroups — lightweight, starts in milliseconds, much higher density per host.
Explain the difference between CMD and ENTRYPOINT.▸
CMD provides default arguments that are fully replaced if the user passes arguments to `docker run`. ENTRYPOINT sets a fixed executable that always runs; any arguments to `docker run` are appended to it rather than replacing it. They're often combined: ENTRYPOINT sets the binary, CMD supplies default flags.
What happens to data written inside a container when it's removed?▸
It's lost — a container's writable layer is deleted along with the container. Any data that must persist has to live in a Docker volume or a bind mount to the host filesystem, which exist independently of the container's lifecycle.
What's the difference between COPY and ADD in a Dockerfile?▸
COPY simply copies files/directories from the build context into the image. ADD does everything COPY does, plus it auto-extracts local tar archives and can fetch files from remote URLs. Best practice is to prefer COPY unless you specifically need ADD's extra behavior, since it's more predictable.
02

Images & Dockerfile

How does Docker's build cache work, and how do you optimize for it?▸
Each instruction produces a layer; Docker reuses a cached layer if the instruction and its inputs are unchanged. Optimize by ordering instructions from least-to-most frequently changing — e.g. copy dependency manifests and install dependencies before copying application source code, so code changes don't invalidate the dependency-install layer.
What is a multi-stage build and why use one?▸
A Dockerfile with multiple FROM stages, where later stages selectively COPY artifacts from earlier ones. It lets you use a full SDK/compiler image to build the app, then ship only the compiled output in a minimal final image — smaller size and a much reduced attack surface.
Why should you avoid using the `latest` tag in production?▸
`latest` is mutable — it can point to a different image tomorrow than it does today, breaking reproducibility and making rollbacks/debugging harder. Pin explicit version tags, or better, image digests (`image@sha256:...`), for immutable, auditable deployments.
How do you reduce Docker image size?▸
Use a minimal base (alpine/distroless), multi-stage builds to drop build tools, combine RUN commands and clean package-manager caches in the same layer, avoid copying unnecessary files (.dockerignore), and remove dev dependencies from the final stage.
03

Networking & Storage

How do two containers on the same host communicate?▸
By joining the same user-defined bridge network, where Docker provides automatic DNS resolution by container name — e.g. an API container can reach a database container simply via hostname `db`, without hardcoding IP addresses.
What's the difference between a volume and a bind mount?▸
A volume is fully managed by Docker (stored under Docker's own directory), portable, and the recommended way to persist data. A bind mount maps an arbitrary host path directly into the container — useful for local development (live-mounting source code) but ties you to the host's filesystem layout.
What does the `host` network mode do, and when would you use it?▸
It removes network isolation, letting the container share the host's network stack directly — no port mapping needed, but also no isolation. Used for cases needing maximum network performance or when a tool truly needs to bind to the host's network directly (rare in normal application deployments).
04

Security & Vulnerability Scanning

Why can deleting a secret file in a later Dockerfile RUN step still leave it recoverable?▸
Image layers are immutable and additive — a later layer that deletes a file only marks it hidden in the merged view; the file's bytes still physically exist in the earlier layer and can be extracted by anyone with access to the image. Secrets must never be baked in at all — use build secrets, runtime env vars, or a secrets manager instead.
What is Trivy and what can it scan besides container images?▸
Trivy is an open-source, all-in-one scanner from Aqua Security. Besides container images, it can scan filesystems, git repositories, Kubernetes clusters, and Infrastructure-as-Code files (Dockerfiles, Terraform, Kubernetes manifests) for vulnerabilities, misconfigurations, exposed secrets, and license issues, and can also generate an SBOM.
How would you scan images in an air-gapped (no internet) environment with Trivy?▸
Download the vulnerability database (trivy-db, distributed as an OCI artifact) on a machine with internet access, transfer the archive to the air-gapped host through an approved offline channel, load it into Trivy's cache directory, then run scans with `--skip-db-update --offline-scan` so no network calls are attempted. The DB needs periodic manual refreshing since it can't auto-update.
Why should containers avoid running as root?▸
If an attacker exploits a vulnerability inside a root-running container, they get root-level access within that container's namespace — which is a stronger foothold and, combined with certain misconfigurations (like a mounted Docker socket or privileged mode), can potentially lead to host-level compromise. Setting a non-root USER in the Dockerfile limits the blast radius.
What's the difference between a vulnerability scan and a misconfiguration/IaC scan?▸
A vulnerability scan checks installed OS/application packages against known CVE databases. A misconfiguration/IaC scan instead checks the *configuration* itself — e.g. a Dockerfile running as root, a Kubernetes pod without resource limits, or an S3 bucket set to public — issues that exist regardless of package versions. Tools like Trivy can do both in one pass.
05

Scenario / Troubleshooting

A container keeps restarting in a crash loop — how do you debug it?▸
Check `docker logs ` for the exit reason first. Run `docker ps -a` to see the exit code, and `docker inspect` for OOMKilled or other state flags. If the entrypoint crashes before logging anything useful, override it temporarily (`docker run -it --entrypoint sh image`) to poke around interactively.
Your image build works locally but fails in CI — what would you check?▸
Common culprits: different Docker/BuildKit version, missing files due to .dockerignore differences, build context not including needed files, platform/architecture mismatch (amd64 vs arm64), or the CI runner lacking cached layers a local machine has, exposing a genuine dependency-resolution issue that was masked by cache locally.
How would you reduce a 1.2GB image down significantly without changing the app?▸
Switch the base image to a slim/alpine or distroless variant, use a multi-stage build so build-time dependencies never reach the final image, remove package manager caches in the same RUN layer they're created in, and check `docker history` to find and eliminate any unexpectedly large layers.
