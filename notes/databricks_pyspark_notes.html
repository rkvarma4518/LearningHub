What is a Spark Cluster?

Foundation · Distributed Computing

A Spark Cluster is a group of machines (nodes) that work together to process large-scale data in a distributed and parallel manner. Instead of one powerful machine, Spark splits work across many machines — making it horizontally scalable.

┌──────────────────────────────────────────────────────────┐ │ SPARK CLUSTER │ └──────────────────────────────────────────────────────────┘ │ ▼ ┌─────────────────────┐ ┌────────────────────────────┐ │ DRIVER NODE │ │ WORKER NODES │ │ ───────────────── │ ──────▶ │ ┌──────────┐ ┌──────────┐ │ │ SparkContext │ │ │ Executor │ │ Executor │ │ │ DAG Scheduler │ │ │ Task 1 │ │ Task 2 │ │ │ Task Scheduler │ │ └──────────┘ └──────────┘ │ └─────────────────────┘ └────────────────────────────┘ │ ▼ Cluster Manager (YARN / Mesos / Kubernetes / Standalone)
🟦

Driver Node

The master process. Runs your Spark application, creates SparkContext, builds DAG, schedules tasks on workers.

🟩

Worker Nodes

Execute tasks assigned by the driver. Each worker has multiple Executors which run tasks in parallel on partitions.

🟧

Cluster Manager

Allocates resources (CPU/memory) to Spark applications. Can be YARN, Kubernetes, Mesos, or Spark Standalone.

🔑 Key Terminology

  • RDD (Resilient Distributed Dataset) — Fundamental data structure; fault-tolerant, distributed collection
  • Partition — A chunk of data processed by a single task on one executor
  • Task — A unit of work sent to one executor for one partition
  • Stage — A group of tasks that can run without a shuffle boundary
  • Job — A full computation triggered by an action (e.g., count(), collect())
  • Executor — JVM process running on a worker, maintaining a thread pool for tasks
🏗️

Spark Architecture

Internals · DAG · Lazy Evaluation

Spark uses a master-worker (driver-executor) architecture with lazy evaluation and DAG-based optimization. Understanding this is key to writing efficient Spark code.

📐 Execution Flow

  • 1. User writes Spark code → transformations are lazy (nothing runs yet)
  • 2. Action is called → Spark starts building an execution plan
  • 3. DAG creation → Logical plan → Physical plan via Catalyst Optimizer
  • 4. DAG Scheduler → Breaks DAG into stages (at shuffle boundaries)
  • 5. Task Scheduler → Sends tasks to available executors on workers
  • 6. Executors run tasks → Results returned to driver or written to storage

🔄 Transformations (Lazy)

Do not trigger execution. They build the DAG.

  • Narrow: filter(), map(), select() — no shuffle
  • Wide: groupBy(), join(), distinct() — trigger shuffle

⚡ Actions (Eager)

Trigger the actual computation.

  • count(), collect(), show(), save()
  • first(), take(n), write.save()
💡
Catalyst Optimizer: Spark's SQL optimizer that rewrites query plans using rule-based and cost-based optimization — pushing down filters, eliminating unused columns, and reordering joins automatically.
✅
Tungsten Engine: Spark's physical execution engine that uses off-heap memory management and code generation (Whole-Stage CodeGen) to produce near-native speed execution.

What is Databricks?

Platform Overview · Lakehouse

Databricks is a Unified Data Analytics Platform built on top of Apache Spark. It was founded by the original creators of Spark and provides a managed, cloud-based environment to run Spark workloads — eliminating infrastructure management.

🎯 What Databricks Provides

  • Managed Apache Spark clusters
  • Collaborative Notebooks (Python, SQL, Scala, R)
  • Delta Lake (ACID transactions on data lakes)
  • MLflow for ML experiment tracking
  • Unity Catalog for governance
  • Databricks SQL for BI workloads
  • Autoloader for incremental data ingestion
  • Workflow orchestration (Jobs)

🏛️ Lakehouse Architecture

  • Combines best of Data Lake (cheap storage) + Data Warehouse (ACID, performance)
  • Data stored as Parquet + Delta transaction log
  • Supports BI, ML, and streaming on same data
  • No data movement needed between systems
  • Open source formats — no vendor lock-in
🏗️
Medallion Architecture (Bronze → Silver → Gold): Raw data lands in Bronze, cleaned/joined data in Silver, business-level aggregates in Gold. Databricks is built around this pattern using Delta Lake tables.

Azure Data Lake Storage (ADLS Gen2)

Storage · Azure Setup

ADLS Gen2 combines the scalability of Azure Blob Storage with a hierarchical namespace (like a file system), enabling efficient big data analytics. It is the primary storage layer for Databricks on Azure.

📦 ADLS Hierarchy

  • Storage Account — Top-level container
  • Container (File System) — Like a bucket or folder root
  • Directory — Hierarchical folder structure
  • Blob/File — Actual data files (CSV, Parquet, JSON)

✅ Why ADLS Gen2?

  • Hierarchical Namespace for atomic directory ops
  • ACL-based fine-grained access control
  • Optimized for Hadoop-compatible workloads
  • Tiered storage (Hot / Cool / Archive)
  • Native integration with Azure services
ABFSS URI FORMAT
# Azure Blob File System Secure (ABFSS) path format
abfss://<container-name>@<storage-account-name>.dfs.core.windows.net/<path>

# Example
abfss://bronze@mydatalake.dfs.core.windows.net/sales/2024/orders.csv

Databricks Spark Cluster Configuration

Cluster Types · Autoscaling · Node Types

🔧 Cluster Types

  • All-Purpose Cluster — For interactive dev, notebooks, shared usage. Manual start/stop.
  • Job Cluster — Spun up for a specific job, terminated after. Cost-efficient for production.
  • SQL Warehouse — Optimized for SQL/BI workloads (Databricks SQL).

📐 Key Config Parameters

  • Cluster Mode: Standard / High Concurrency / Single Node
  • Databricks Runtime: Version with Spark + Delta + ML libs
  • Worker Type: VM size (e.g., Standard_DS3_v2)
  • Min/Max Workers: For autoscaling range
  • Autoscaling: Adds/removes workers based on load
  • Auto Termination: Idle timeout (e.g., 20 mins)
⚠️
Cost Tip: Always enable auto-termination on All-Purpose clusters during development. Job clusters are cheaper for production since they terminate automatically after the job finishes.

Databricks File System (DBFS)

Storage Abstraction · Mount Points

DBFS is a distributed file system abstraction layer built into Databricks. It abstracts cloud storage (ADLS, S3, GCS) and provides a consistent file path syntax across all cloud providers.

📋 DBFS Key Points

  • All Databricks notebooks can reference paths using dbfs:/ prefix or just /
  • Files stored in DBFS are actually stored in the cloud storage account backing your workspace
  • The /FileStore folder is the default upload location (accessible via UI)
  • Mount points map cloud paths → DBFS paths for easy access
  • Not recommended for production data — use direct ABFSS or mount points to ADLS instead
DBFS — PYTHON
# Using dbutils to interact with DBFS
dbutils.fs.ls("dbfs:/FileStore/")
dbutils.fs.ls("/mnt/")  # List mount points

# Copy file within DBFS
dbutils.fs.cp("dbfs:/FileStore/data.csv", "dbfs:/mnt/bronze/data.csv")

# Remove a file or folder
dbutils.fs.rm("dbfs:/tmp/old_data/", recurse=True)

# Read from DBFS in PySpark
df = spark.read.csv("dbfs:/FileStore/orders.csv", header=True)

Access Azure Data Lake from Databricks

Authentication · Mount Points · ABFSS

There are three main ways to access ADLS from Databricks, each with different security implications:

  1. 1

    Storage Account Key (Quick & Dirty)

    Directly set the storage account key in Spark config. NOT recommended for production — exposes the key.

  2. 2

    SAS Token (Scoped Access)

    Use a Shared Access Signature token — time-limited and scoped to specific resources. Better, but tokens need rotation.

  3. 3

    Service Principal (Recommended)

    Register an app in Azure AD, grant it Storage Blob Data Contributor role on ADLS, use OAuth 2.0. Best practice for production — stored in Key Vault Secrets.

METHOD 1 — STORAGE KEY (DEV ONLY)
spark.conf.set(
    "fs.azure.account.key.<storage-account>.dfs.core.windows.net",
    "<your-storage-account-key>"
)

df = spark.read.csv(
    "abfss://bronze@<storage-account>.dfs.core.windows.net/data.csv",
    header=True
)
METHOD 3 — SERVICE PRINCIPAL (PRODUCTION)
# Set Spark config for OAuth authentication
spark.conf.set("fs.azure.account.auth.type.<storage>.dfs.core.windows.net", "OAuth")
spark.conf.set("fs.azure.account.oauth.provider.type.<storage>.dfs.core.windows.net",
               "org.apache.hadoop.fs.azurebfs.oauth2.ClientCredsTokenProvider")
spark.conf.set("fs.azure.account.oauth2.client.id.<storage>.dfs.core.windows.net",
               client_id)
spark.conf.set("fs.azure.account.oauth2.client.secret.<storage>.dfs.core.windows.net",
               client_secret)
spark.conf.set("fs.azure.account.oauth2.client.endpoint.<storage>.dfs.core.windows.net",
               f"https://login.microsoftonline.com/{tenant_id}/oauth2/token")

Service Principal in Azure

Azure AD · App Registration · IAM

A Service Principal is the identity a non-human application uses to access Azure resources. Think of it as a "machine user" with specific permissions.

  1. 1

    Register an App in Azure Active Directory

    Azure Portal → Azure Active Directory → App Registrations → New Registration. Note the Application (client) ID and Tenant ID.

  2. 2

    Create a Client Secret

    Under the registered app → Certificates & Secrets → New Client Secret. Copy the secret value (only shown once!).

  3. 3

    Grant IAM Role on ADLS

    Storage Account → IAM → Add Role Assignment → "Storage Blob Data Contributor" → select your app.

  4. 4

    Store Secrets in Key Vault

    Never hardcode credentials. Store client_id, client_secret, tenant_id in Azure Key Vault. Link Key Vault to Databricks Secret Scope.

📋 Three Values Needed for SP Auth

  • client_id — Application (client) ID from App Registration
  • client_secret — Secret generated under Certificates & Secrets
  • tenant_id — Directory (tenant) ID from Azure AD Overview

Reading Data (Ingestion) using PySpark

DataFrameReader · File Formats · Schema

PySpark's DataFrameReader API reads data from virtually any source into a DataFrame. Reading is a transformation — actual data is not loaded until an action is triggered.

PYSPARK — READ CSV / JSON / PARQUET
# CSV with options
df_csv = spark.read \
    .option("header", True) \
    .option("inferSchema", True) \
    .option("sep", ",") \
    .csv("/mnt/bronze/orders.csv")

# JSON — single line and multiline
df_json = spark.read \
    .option("multiline", True) \
    .json("/mnt/bronze/events.json")

# Parquet (schema auto-inferred)
df_parquet = spark.read.parquet("/mnt/silver/customers/")

# Delta table
df_delta = spark.read.format("delta").load("/mnt/gold/sales/")
df_sql   = spark.read.table("gold.sales")

# Explicit schema (best practice for production)
from pyspark.sql.types import StructType, StructField, StringType, IntegerType

schema = StructType([
    StructField("order_id",   IntegerType(), True),
    StructField("customer",   StringType(),  True),
    StructField("amount",      IntegerType(), True),
])

df = spark.read.schema(schema).csv("/mnt/bronze/orders.csv", header=True)
⚠️
inferSchema warning: inferSchema=True reads data twice (one pass to infer types, one to load). For large datasets, always define an explicit schema to save time and avoid wrong type inference.

Data Transformation using PySpark

DataFrame API · SQL Functions · Window
CORE TRANSFORMATIONS
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# Select, rename, cast columns
df = df.select(
    F.col("order_id").alias("id"),
    F.col("amount").cast("double"),
    F.upper(F.col("customer_name")).alias("customer")
)

# Filter rows
df = df.filter(F.col("amount") > 100)
df = df.where((F.col("status") == "ACTIVE") & (F.col("year") == 2024))

# Add/derive new columns
df = df.withColumn("vat_amount", F.col("amount") * 0.18)
df = df.withColumn("load_date", F.current_date())
df = df.withColumn("full_name", F.concat_ws(" ", F.col("first"), F.col("last")))

# Group by and aggregate
df_agg = df.groupBy("category", "region").agg(
    F.sum("amount").alias("total_sales"),
    F.avg("amount").alias("avg_sales"),
    F.count("order_id").alias("order_count"),
    F.max("amount").alias("max_order")
)

# Joins
df_joined = df_orders.join(df_customers, on="customer_id", how="left")

# Window functions — rank by sales per region
window_spec = Window.partitionBy("region").orderBy(F.desc("amount"))
df = df.withColumn("rank", F.rank().over(window_spec))
df = df.withColumn("running_total", F.sum("amount").over(window_spec))

Real-World PySpark Scenarios

Production Patterns · Performance

📌 Handling Duplicates

  • df.dropDuplicates(["order_id"]) — drop exact dupes on key columns
  • Window function with row_number to pick the latest record per key

📌 Handling Nulls

  • df.dropna(subset=["amount"])
  • df.fillna({"amount": 0, "status": "UNKNOWN"})
  • Use F.coalesce() for first non-null value

📌 Skew Handling

  • Broadcast smaller DataFrames in joins: F.broadcast(df_small)
  • Salting technique for hot keys in groupBy
  • Repartition by a balanced column before joining

📌 Partitioning Strategy

  • df.repartition(n) — full shuffle, creates n equal partitions
  • df.coalesce(n) — reduce partitions without full shuffle
  • df.write.partitionBy("year","month") — partition on disk
DEDUPLICATION — LATEST RECORD PER KEY
from pyspark.sql.window import Window
from pyspark.sql import functions as F

window = Window.partitionBy("order_id").orderBy(F.desc("updated_at"))
df = df.withColumn("row_num", F.row_number().over(window))
df_deduped = df.filter(F.col("row_num") == 1).drop("row_num")

Delta Lake Architecture

ACID · Parquet + Log · Lakehouse

Delta Lake is an open-source storage layer that brings ACID transactions, schema enforcement, and time travel to data lakes. It stores data as Parquet files with a transaction log (_delta_log) on top.

Delta Table = Parquet Files + _delta_log/ ┌────────────────────────────────────────────────────────────┐ │ /mnt/delta/orders/ │ ├────────────────────────────────────────────────────────────┤ │ 📄 part-00000.parquet 📄 part-00001.parquet │ │ 📄 part-00002.parquet 📄 part-00003.parquet │ │ │ │ 📁 _delta_log/ │ │ 00000000000000000000.json (version 0 - initial) │ │ 00000000000000000001.json (version 1 - insert) │ │ 00000000000000000002.json (version 2 - update) │ │ 00000000000000000010.checkpoint.parquet │ └────────────────────────────────────────────────────────────┘
🔒

ACID Transactions

Atomicity, Consistency, Isolation, Durability — concurrent reads and writes are safe.

⏪

Time Travel

Query any previous version of the data using version number or timestamp.

📐

Schema Enforcement

Rejects bad data at write time. Schema evolution also supported with mergeSchema.

⚡

Scalable Metadata

Delta log handles millions of files efficiently — no bottleneck like traditional Hive Metastore.

🔄

Upserts / Merges

Native MERGE INTO support — update, insert, or delete rows based on conditions.

📊

Statistics & Data Skipping

Delta stores min/max statistics per file — Spark skips irrelevant files automatically.

CRUD Operations in Delta Tables

Insert · Update · Delete · Merge
INSERT / UPDATE / DELETE
-- INSERT into Delta table INSERT INTO silver.orders SELECT * FROM bronze.orders WHERE status = 'NEW'; -- UPDATE rows UPDATE silver.orders SET status = 'PROCESSED', updated_at = current_timestamp() WHERE order_date < '2024-01-01'; -- DELETE rows DELETE FROM silver.orders WHERE status = 'CANCELLED' AND order_date < '2023-01-01';
MERGE INTO (UPSERT) — MOST IMPORTANT PATTERN
MERGE INTO silver.orders AS target USING bronze.orders_updates AS source ON target.order_id = source.order_id WHEN MATCHED AND source.updated_at > target.updated_at THEN UPDATE SET target.amount = source.amount, target.status = source.status, target.updated_at = source.updated_at WHEN NOT MATCHED THEN INSERT (order_id, customer_id, amount, status, updated_at) VALUES (source.order_id, source.customer_id, source.amount, source.status, source.updated_at) WHEN NOT MATCHED BY SOURCE THEN DELETE;
MERGE — PYSPARK API
from delta.tables import DeltaTable delta_table = DeltaTable.forPath(spark, "/mnt/silver/orders") delta_table.alias("target").merge( source=df_updates.alias("source"), condition="target.order_id = source.order_id" ).whenMatchedUpdateAll() \ .whenNotMatchedInsertAll() \ .execute()

Delta Log Explained in Detail

Transaction Log · Checkpoints · Metadata

The _delta_log folder is the brain of Delta Lake. Every write operation appends a new JSON file (a "commit") to this log, recording exactly what changed — which files were added, which were removed, and what the schema looks like.

📄 What's in Each Delta Log Entry (JSON)

  • commitInfo — Operation type (WRITE, UPDATE, DELETE, MERGE), timestamp, user, parameters
  • add — New Parquet files added with stats (min, max, nullCount per column)
  • remove — Files marked as deleted (soft-delete — file still physically exists until VACUUM)
  • metaData — Schema, partitioning info, Delta table configuration
  • protocol — Minimum reader/writer version required
DELTA TABLE HISTORY & LOG
# View transaction history from delta.tables import DeltaTable dt = DeltaTable.forPath(spark, "/mnt/silver/orders") dt.history(10).show(truncate=False) -- SQL equivalent DESCRIBE HISTORY silver.orders; -- Detail shows table metadata DESCRIBE DETAIL silver.orders;
📌
Checkpoint files: Every 10 commits, Delta creates a .checkpoint.parquet file that consolidates all previous JSON logs. This speeds up reading history — Spark only needs to read the latest checkpoint + subsequent JSON files rather than all JSON logs from version 0.
⏪

Time Travel & VACUUM

Data Versioning · Rollback · Cleanup

Delta Lake keeps all previous versions of data by default. Time Travel lets you query past states. VACUUM physically deletes old data files to reclaim storage.

TIME TRAVEL
# Read a specific version df_v2 = spark.read.format("delta") \ .option("versionAsOf", 2) \ .load("/mnt/silver/orders") # Read at a specific timestamp df_ts = spark.read.format("delta") \ .option("timestampAsOf", "2024-06-01 12:00:00") \ .load("/mnt/silver/orders") -- SQL syntax SELECT * FROM silver.orders VERSION AS OF 3; SELECT * FROM silver.orders TIMESTAMP AS OF '2024-06-01'; # Restore to previous version dt = DeltaTable.forPath(spark, "/mnt/silver/orders") dt.restoreToVersion(5) # or: dt.restoreToTimestamp("2024-06-01 12:00:00")
VACUUM — RECLAIM STORAGE
-- Default: remove files older than 7 days (168 hours) VACUUM silver.orders; -- Custom retention (minimum 168 hours recommended) VACUUM silver.orders RETAIN 240 HOURS; -- DRY RUN — shows what would be deleted VACUUM silver.orders DRY RUN; -- Force vacuum below 7-day limit (DANGER: breaks time travel!) spark.conf.set("spark.databricks.delta.retentionDurationCheck.enabled", False) VACUUM silver.orders RETAIN 0 HOURS;
⚠️
VACUUM Warning: After running VACUUM, you cannot time-travel to versions older than the retention period. Set retention ≥ your longest streaming query checkpoint lag to avoid breaking streaming jobs.

Delta Table Optimization & ZORDER BY

Compaction · File Layout · Data Skipping

🔧 OPTIMIZE Command

Delta tables accumulate many small Parquet files over time (from streaming or frequent writes). OPTIMIZE compacts them into fewer, larger files — improving query performance and reducing metadata overhead.

OPTIMIZE & ZORDER
-- Basic optimize (file compaction) OPTIMIZE silver.orders; -- Optimize a partition OPTIMIZE silver.orders WHERE date >= '2024-01-01'; -- OPTIMIZE with ZORDER BY (colocates similar values) OPTIMIZE silver.orders ZORDER BY (customer_id, order_date);

🧲 How ZORDER BY Works

  • ZORDER is a multi-dimensional locality-preserving space-filling curve that rearranges data within Parquet files so that records with similar column values are stored together
  • When you query WHERE customer_id = 'C123', Spark can skip entire files that don't contain that customer
  • Works best on high-cardinality columns used in WHERE filters (e.g., user_id, product_id, region)
  • Use on 2–3 columns max — diminishing returns beyond that
  • Do NOT ZORDER by partitioning column — already sorted at partition level
💡
Auto Optimize (Databricks feature): Enable delta.autoOptimize.optimizeWrite = true and delta.autoOptimize.autoCompact = true on a table to automatically compact files during writes — no manual OPTIMIZE needed.
AUTO OPTIMIZE SETTINGS
-- Enable auto optimize on a table ALTER TABLE silver.orders SET TBLPROPERTIES ( 'delta.autoOptimize.optimizeWrite' = 'true', 'delta.autoOptimize.autoCompact' = 'true' );

Autoloader (cloudFiles)

Incremental Ingestion · File Notification

Autoloader is Databricks' built-in mechanism for incremental data ingestion from cloud storage. It automatically detects and processes new files as they arrive — without you needing to track which files have been processed.

✅ Why Autoloader?

  • No need to list all files manually every run
  • Tracks processed files in a checkpoint
  • Scales to millions of files with file notification mode
  • Handles schema inference and evolution automatically
  • Native integration with Structured Streaming

🔔 Two Discovery Modes

  • Directory Listing — Default; lists all files and filters new ones. Simple but slower at scale.
  • File Notification — Uses Azure Event Grid / AWS SNS to get notified of new files. Scales to billions of files.
AUTOLOADER — INCREMENTAL INGESTION
# Read new files as they arrive df_stream = spark.readStream.format("cloudFiles") \ .option("cloudFiles.format", "csv") \ .option("cloudFiles.schemaLocation", "/mnt/checkpoints/orders_schema") \ .option("header", "true") \ .load("/mnt/bronze/incoming_orders/") # Add metadata columns df_stream = df_stream.withColumn("source_file", F.input_file_name()) \ .withColumn("ingestion_time", F.current_timestamp()) # Write to Delta table query = df_stream.writeStream \ .format("delta") \ .outputMode("append") \ .option("checkpointLocation", "/mnt/checkpoints/orders_load") \ .trigger(availableNow=True) \ # Process all available, then stop .start("/mnt/silver/orders/") query.awaitTermination()
💡
trigger(availableNow=True) processes all available files in micro-batches then stops — perfect for scheduled batch-style runs with Autoloader. Use trigger(processingTime="5 minutes") for continuous near-real-time streaming.

Structured Streaming in Databricks

Kafka · Event Hubs · Delta Streaming

Structured Streaming treats a live data stream as an unbounded table — you write the same DataFrame transformations you'd use for batch, and Spark handles the streaming mechanics.

KAFKA → DELTA STREAMING PIPELINE
# Read from Kafka (or Azure Event Hubs) df_kafka = spark.readStream \ .format("kafka") \ .option("kafka.bootstrap.servers", "broker1:9092,broker2:9092") \ .option("subscribe", "orders_topic") \ .option("startingOffsets", "latest") \ .load() # Parse JSON payload from Kafka value schema = StructType([ StructField("order_id", StringType()), StructField("amount", DoubleType()), StructField("event_time", TimestampType()) ]) df_parsed = df_kafka \ .select(F.from_json(F.col("value").cast("string"), schema).alias("data")) \ .select("data.*") # Apply watermarking for late data handling df_windowed = df_parsed \ .withWatermark("event_time", "10 minutes") \ .groupBy(F.window("event_time", "5 minutes"), "order_id") \ .agg(F.sum("amount").alias("total")) # Write to Delta df_windowed.writeStream \ .format("delta") \ .outputMode("append") \ .option("checkpointLocation", "/mnt/checkpoints/kafka_orders") \ .start("/mnt/gold/orders_agg/")

🔑 Key Streaming Concepts

  • Checkpoint: Stores stream progress — enables restart from where it left off
  • Watermark: Threshold for how late data can arrive — enables stateful operations in streaming
  • Trigger: How often to run micro-batches (ProcessingTime / Once / AvailableNow / Continuous)
  • Fault Tolerance: Exactly-once semantics using checkpoints + idempotent sinks (Delta Lake)

Databricks Workflows & Jobs

Orchestration · Scheduling · DAG

Databricks Workflows is the built-in orchestration tool. Create multi-task pipelines (DAGs) where tasks can be Notebooks, Python scripts, SQL queries, JAR files, or dbt projects.

⚙️ Job Configuration

  • Cluster: Use a Job Cluster (created and terminated per run) for cost savings
  • Schedule: Cron expression (e.g., 0 6 * * * for 6AM daily)
  • Parameters: Pass values to notebooks via widget parameters
  • Alerts: Email / Slack notifications on success/failure
  • Retry Policy: Auto-retry failed tasks N times with delay
  • Timeout: Kill job after specified duration

📋 Multi-Task Workflow Pattern

  • Task 1: Ingest raw data (Bronze) — Autoloader
  • Task 2: Clean + validate (Silver) — depends on Task 1
  • Task 3a, 3b: Parallel aggregations (Gold)
  • Task 4: Notify/export downstream — depends on 3a, 3b
[Trigger: Cron 6AM] │ [Task 1: Autoloader Ingest] │ [Task 2: Silver Transformation] ┌──┴──┐ [3a: Sales] [3b: Inventory] ← parallel tasks └──┬──┘ [Task 4: Export to BI / Notify]
💡
Job Cluster vs All-Purpose Cluster: Always use a Job Cluster for production Workflow runs. It spins up fresh, runs the job, then terminates — you only pay for what you use. All-Purpose clusters stay up and cost money even when idle.

Azure End-To-End Data Engineering Project

Architecture · Medallion · Production

A complete production data pipeline on Azure using Databricks follows the Medallion Architecture:

[Source Systems] REST APIs / SQL DBs / Files / Kafka │ ▼ ┌──────────────────────────────┐ │ BRONZE LAYER (Raw) │ ← ADLS Container: bronze/ │ Autoloader / ADF / Kafka │ ← Parquet / Delta, as-is │ No transformation │ └──────────────────────────────┘ │ ▼ ┌──────────────────────────────┐ │ SILVER LAYER (Cleansed) │ ← ADLS Container: silver/ │ Dedup, Null handling │ ← Delta Tables with schema │ Type casting, Joins │ ← SCD Type 2 history └──────────────────────────────┘ │ ▼ ┌──────────────────────────────┐ │ GOLD LAYER (Business) │ ← ADLS Container: gold/ │ Aggregations, KPIs │ ← Optimized for BI queries │ Fact & Dimension tables │ └──────────────────────────────┘ │ ▼ [Consumers] Power BI / Synapse SQL / ML Models / APIs
  1. 1

    Infrastructure Setup

    Create Resource Group → ADLS Gen2 (bronze, silver, gold containers) → Databricks Workspace → Azure Key Vault (store SP credentials).

  2. 2

    Authentication

    Register Service Principal in Azure AD → Grant Storage Blob Data Contributor on ADLS → Store client_id, client_secret, tenant_id in Key Vault → Create Databricks Secret Scope backed by Key Vault.

  3. 3

    Bronze Layer — Raw Ingestion

    Use Autoloader with cloudFiles format to ingest CSVs/JSON from landing zone into bronze Delta tables. Add metadata (source_file, ingestion_timestamp). No data transformation.

  4. 4

    Silver Layer — Cleansing & Enrichment

    Deduplicate using window functions, handle nulls, cast data types, join dimension tables, apply SCD Type 2 for historical tracking. Write as Delta tables with MERGE INTO for upserts.

  5. 5

    Gold Layer — Business Aggregations

    Create fact and dimension tables, compute KPIs (daily sales, customer lifetime value, inventory levels). OPTIMIZE + ZORDER for fast BI queries.

  6. 6

    Orchestration

    Create Databricks Workflow with multi-task Job. Schedule daily at 6AM. Bronze → Silver → Gold tasks in sequence. Alert on failure via email. Use Job Clusters for cost efficiency.

  7. 7

    Consumption

    Connect Power BI to Databricks SQL Warehouse for dashboards. Expose Gold tables via REST API or Azure Synapse Serverless SQL. ML models read from Silver/Gold.

🏆 Production Best Practices

  • Always use Service Principal auth — never hardcode storage keys
  • Explicit schemas — never rely on inferSchema in production
  • External Delta tables for all production data — data survives if catalog is dropped
  • Checkpoint every streaming query — enables restarts without data loss
  • OPTIMIZE + ZORDER after bulk loads or daily ingestion
  • Schedule VACUUM weekly with 7+ day retention
  • Use Job Clusters for all production runs — not All-Purpose
  • Enable Auto Optimize on high-write tables (streaming sinks)
  • Monitor with Databricks SQL Dashboard or Azure Monitor
  • Unity Catalog for fine-grained row/column level access control
